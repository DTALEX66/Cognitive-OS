## 一、系统定位

### 1.1 MCP扩展在Agent系统中的角色

MCP(Model Context Protocol)扩展系统是B-Line多Agent协作架构中的**工具桥梁**，承担着将外部工具和服务无缝接入Agent生态系统的核心职责。在整个B-Line架构中，MCP扩展系统不直接参与Agent的决策或执行，而是**作为一个标准化的工具接入层，为所有Agent提供统一的外部能力调用接口**。

MCP扩展系统在B-Line架构中的角色体现在三个维度：

**1. 工具发现者** —— Agent通过MCP协议发现和探索外部工具的能力。MCP提供标准化的工具清单、参数描述和调用方式，Agent无需了解工具的内部实现即可使用。

**2. 协议转换器** —— MCP将外部工具的各种通信协议(HTTP、WebSocket、子进程等)统一为Agent可以理解的标准化接口。Agent不关心工具是通过什么协议暴露的，它只看到一个统一的函数调用界面。

**3. 能力边界扩展者** —— MCP是Agent能力的延伸。通过MCP，Agent可以访问搜索引擎、翻译服务、代码分析工具、文档抓取服务等多种外部能力，大大拓展了Agent的功能边界。

### 1.2 核心设计原则

**原则一：协议统一** —— 无论底层使用什么传输方式，Agent看到的MCP工具接口是一致的。统一的工具发现、调用、结果返回协议。

**原则二：即插即用** —— 新的MCP工具应该能够在不修改Agent代码的情况下注册和使用。工具的自描述元数据(名称、参数、返回值)就是Agent使用工具所需的全部信息。

**原则三：安全嵌套** —— MCP工具调用必须嵌套在05权限系统之下。任何MCP工具调用都不能绕过权限校验。

**原则四：优雅降级** —— 当MCP工具不可用或超时时，系统应有明确的降级策略。不能因一个工具的故障拖垮整个Agent工作流。

**原则五：可观测性** —— 所有MCP工具调用应当有完整的生命周期追踪：发起、连接、调用、返回、超时、错误。

## 二、传输方式对比

### 2.1 四种传输方式总览

MCP标准定义了四种传输方式，每种方式适合不同的使用场景。B-Line 06将这四种方式统一纳入Agent的工具接入层。

| 传输方式 | 通信模型 | 连接类型 | 适用场景 |
|---------|---------|---------|---------|
| Stdio(标准输入输出) | 请求-响应 | 子进程管道 | 本地工具、脚本 |
| SSE(Server-Sent Events) | 单向推送+请求-响应 | HTTP长连接 | 远程API、监控 |
| StreamableHTTP | 流式请求-响应 | HTTP流 | 大文件、流式结果 |
| WebSocket | 双向全双工 | TCP长连接 | 实时交互、推送 |

### 2.2 StdioClientTransport - 标准输入输出

Stdio是最常用的MCP传输方式，通过子进程的标准输入(stdin)和标准输出(stdout)进行通信。Agent向子进程的stdin发送JSON-RPC请求，从stdout读取响应。

**工作原理**：Agent启动子进程(如python脚本、node程序、二进制文件)，通过stdin发送JSON-RPC格式的工具调用请求，子进程处理请求并通过stdout返回JSON-RPC格式的响应。

**优点**：实现简单、无需网络配置、延迟低、安全性好(子进程隔离)、适合集成命令行工具。

**缺点**：子进程生命周期管理复杂、不适合远程服务、进程间通信受OS限制。

**典型使用场景**：本地代码分析工具(linter、formatter)、文件处理脚本、本地数据库查询。

### 2.3 SSEClientTransport - Server-Sent Events

SSE是一种基于HTTP的单向推送协议，由服务端向客户端推送事件流。客户端的请求通过独立的HTTP POST完成。

**工作原理**：Agent向MCP服务端发起HTTP连接建立SSE事件流，服务端通过SSE流推送事件通知，Agent通过HTTP POST端点发送工具调用请求。

**优点**：基于标准HTTP协议、防火墙友好、支持服务端主动推送、实现相对简单。

**缺点**：单向推送、不适用于高频双向通信、连接数多时服务端资源消耗大。

**典型使用场景**：远程API服务(搜索、翻译)、需要服务端推送通知的工具。

### 2.4 StreamableHTTPClientTransport - 流式HTTP

StreamableHTTP是SSE的增强版本，支持流式请求和响应。客户端可以发送流式请求，服务端可以返回流式结果。

**工作原理**：Agent通过HTTP POST发起流式请求，服务端逐步处理并以chunked encoding方式逐步返回响应。

**优点**：支持大文件流式传输、支持增量结果实时展现、降低首字节延迟。

**缺点**：实现复杂度高于普通SSE、流中断恢复复杂、服务端架构要求较高。

**典型使用场景**：大文件处理、AI推理调用(逐token返回)、长时间运行的数据分析。

### 2.5 WebSocketTransport - WebSocket

WebSocket提供全双工通信，客户端和服务端可在同一连接上相互发送消息。

**工作原理**：Agent通过HTTP Upgrade建立WebSocket连接，双方通过帧双向发送JSON-RPC消息，均可主动发起请求和推送通知。

**优点**：真正的全双工通信、低延迟、适合高频交互、支持服务端主动推送。

**缺点**：实现复杂度最高、需要WebSocket-aware代理、某些企业网络限制。

**典型使用场景**：实时协作工具、高频数据流处理、需要双向推送的交互式工具。

### 2.6 传输方式选择指南

1. **本地工具优先使用Stdio**：工具与Agent同机时最简单可靠。
2. **远程API优先使用SSE**：标准化程度最高，防火墙兼容性最好。
3. **大文件或流式结果使用StreamableHTTP**：需要传输大文件或实时展示结果时。
4. **高频双向通信使用WebSocket**：需要频繁双向交互时。

## 三、工具注册

### 3.1 工具发现机制

MCP的工具发现通过标准化的初始化握手完成。Agent与MCP服务建立连接后发送initialize请求，服务端返回能力声明和工具列表。然后发送tools/list获取完整清单。每个工具包含：name(唯一标识)、description(用途描述)、inputSchema(JSON Schema参数描述)。

### 3.2 工具注册流程

步骤一：服务声明(MCP服务配置文件声明基本信息)。步骤二：连接建立(根据传输方式建立连接)。步骤三：初始化握手(发送initialize请求)。步骤四：工具列表获取(发送tools/list)。步骤五：注册到全局工具注册表。步骤六：通过事件路由通知Agent层新工具可用。

### 3.3 工具注册表结构

注册表维护所有已注册MCP工具元数据：toolId、name、description、inputSchema、mcpServerId、transportType、status(active/inactive/error)、permissionLevel(L1/L2/L3)、rateLimit(限流配置)、timeout(调用超时毫秒)。

### 3.4 工具生命周期管理

注册(自动或手动) -> 激活(默认active) -> 去激活(维护时暂停) -> 重新激活 -> 卸载。期间通过ping监测可用性，状态变更通过事件路由通知。

## 四、与05权限系统的对接

### 4.1 对接设计

MCP工具调用必须经过05权限系统完整六层校验，不能绕过。校验流程：Zod校验参数 -> Tool自检 -> Pre-Hooks(限流/认证) -> canUseTool权限裁决 -> 通过MCP传输层执行 -> Post-Hooks审计记录。

### 4.2 MCP工具权限级别映射

web_search(L1只读)、fetch(L1只读)、translate(L1只读)、lsp(L1只读)、写文件MCP(L2写入)、数据库写入MCP(L2写入)、删除MCP(L3危险)、执行MCP(L3危险)。

### 4.3 安全校验增强

MCP特有安全检查：认证检查(确保认证有效)、出站安全(SSRF防护)、数据脱敏、配额检查(API调用次数限制)、隔离执行(避免恶意工具影响系统)。

### 4.4 审计日志对接

MCP调用审计增加字段：mcpServerId、transportType、toolName、callDuration(含网络延迟)、responseSize、errorType。与05标准日志一起存储，统一查询。

## 五、可扩展的工具生态

### 5.1 开放式MCP接口设计

基于JSON-RPC 2.0协议，工具自描述(通过inputSchema声明参数)，自动发现(通过tools/list)，支持多种结果格式(文本、图片、资源)。任何遵循该规范的外部工具都可无缝接入。

### 5.2 工具生态分类

信息获取工具：搜索(web_search)、文档抓取(fetch)、知识库查询。处理分析工具：翻译(translate)、代码分析(lsp)、文本处理。交互操作工具：浏览器控制(browser)、文件操作、数据库操作。监控诊断工具：系统状态、日志分析、性能监控。AI增强工具：文本生成、图像生成、代码生成。

### 5.3 第三方工具接入流程

实现MCP协议(官方SDK：TypeScript/Python/Java) -> 选择传输方式 -> 编写工具定义(name + description + inputSchema) -> 配置接入B-Line系统 -> 测试验证 -> 发布到工具市场。

### 5.4 工具市场设计理念

发现(浏览搜索工具)、评级(用户评价辅助质量评估)、版本管理(版本锁定)、安全审核(提交需审核)、使用统计(调用次数、成功率、延迟)等核心功能。

### 5.5 内置MCP工具参考

web_search MCP：Stdio传输，L1只读，参数query+max_results。translate MCP：SSE传输，L1只读，参数text+source_lang+target_lang。lsp MCP：Stdio传输，L1只读，参数file_path+analysis_type。fetch MCP：SSE传输，L1只读，参数url+max_size。

4. 开放式工具生态可无限扩展
