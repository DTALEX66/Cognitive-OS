> 子系统编号：01-Coordinator-from-CW  
> 版本：v1.0  
> 状态：设计文档

## 1. 系统定位

### 1.1 协调器在多 Agent 系统中的核心角色

B-Line 协调器（Coordinator）是整个多 Agent 系统的**神经中枢**。在 CW（Claw Code）的原始设计中，OmX 工作流编排引擎负责生成高层指令流，而 Coordinator 负责将这些指令分解为可执行的 Agent 子任务，并监督其完成。在 B-Line 的上下文中，Coordinator 的角色被进一步明确为以下几个关键职责：

- **任务接收与解析**：从 OmX（上层工作流引擎）接收任务指令，解析其语义、范围和约束条件。
- **任务分解（Decomposition）**：将大型、复杂的任务指令拆解为粒度适合单个 Agent 执行的子任务单元。每个子任务应具备明确的输入、输出、成功标准和上下文边界。
- **Agent 分配与调度**：根据子任务的性质、依赖关系和资源需求，选择合适的 Agent（DEEP 实例）进行分配。Agent 选择策略可基于能力匹配、负载均衡、亲和性调度等多种算法。
- **执行监控与状态跟踪**：实时跟踪每个 Agent 的任务执行状态（pending、running、completed、failed、cancelled），并在异常发生时采取恢复措施。
- **结果归并（Merge）**：收集所有 Agent 的输出结果，按照预定义的合并策略将其整合为完整的、一致的最终输出。
- **事件上报**：通过 clawhip 事件路由层，将协调结果通知下游系统（Discord、Git、CI 流水线等）。

### 1.2 从手动触发到自动编排的升级路径

当前 CODEX 的工作模式以**手动触发**为主：用户手动启动 CODEX 会话，手动指定 DEEP 的任务序列，手动检查执行结果。B-Line 协调器的引入，将彻底改变这一模式：

| 维度 | 当前（手动） | 目标（自动编排） |
|------|------------|----------------|
| 触发方式 | 用户手动启动 | OmX 自动编排触发 |
| 任务分配 | 用户手动指定 Agent | Coordinator 自动分配 |
| 执行顺序 | 串行执行 | 并行 + 依赖图驱动 |
| 错误处理 | 人工介入 | 自动重试 / 降级 / 回滚 |
| 结果整合 | 手动 Copy-Paste | 自动归并 |
| 通知 | 无 | clawhip → Discord / CI |

### 1.3 与 A-Line 的关系

A-Line 定义了 B-Line 的架构约束和接口契约。协调器系统遵循 A-Line 的以下规范：

- **03-Plan-Revision-from-CC-TaskSystem**：任务模型使用 CC Task 格式，协调器需遵循 Task 的生命周期管理规范。
- **通信协议**：协调器与 Agent 之间的通信遵循 A-Line 定义的 MCP 接口规范。
- **安全模型**：协调器自身不执行操作，仅做编排；所有写操作必须经由 Agent 权限层（05-Agent-Permission-from-CC）。

## 2. 架构设计

### 2.1 三层架构：OmX → Coordinator → Agent

B-Line 协调器系统采用严格的三层架构，层与层之间通过明确定义的接口通信：

```
┌──────────────────────────────────────────────────────┐
│                     OmX (工作流编排)                    │
│  宏观策略制定 / 工作流定义 / 业务规则 / 定时触发器     │
└──────────────────────┬───────────────────────────────┘
                       │ 指令下发 (JSON / YAML)
                       ▼
┌──────────────────────────────────────────────────────┐
│                  Coordinator (主协调器)                  │
│  任务分解 / Agent 调度 / 结果归并 / 状态管理 / 错误恢复 │
└──────┬──────────┬──────────┬──────────┬──────────────┘
       │          │          │          │
       ▼          ▼          ▼          ▼
┌─────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────────┐
│Agent│ │Agent │ │Agent │ │Agent │ │   ...    │
│  A  │ │  B   │ │  C   │ │  D   │ │ Agent N  │
└─────┘ └──────┘ └──────┘ └──────┘ └──────────┘
```

**层职责说明**：

| 层级 | 组件 | 职责范围 | 运行形态 |
|------|------|---------|---------|
| L1 | OmX | 工作流定义、业务编排、触发条件 | CW 进程，常驻 |
| L2 | Coordinator | 任务分解、Agent 调度、结果归并 | CODEX 实例，按需启动 |
| L3 | Agent | 具体任务执行、工具调用 | DEEP 实例，按需启动 |

### 2.2 Coordinator 内部模块设计

Coordinator 本身由以下几个内部模块组成：

```
Coordinator 内部结构
┌─────────────────────────────────────────────────────┐
│  Coordinator Runtime                                 │
│  ┌──────────┐  ┌──────────┐  ┌──────────────────┐  │
│  │ 分解引擎  │  │ 调度引擎  │  │ 归并引擎(合并器) │  │
│  └──────────┘  └──────────┘  └──────────────────┘  │
│  ┌──────────┐  ┌──────────┐  ┌──────────────────┐  │
│  │ 状态管理  │  │ 错误恢复  │  │ 事件发布器       │  │
│  └──────────┘  └──────────┘  └──────────────────┘  │
│  ┌─────────────────────────────────────────────┐    │
│  │  Agent 注册中心 / 能力目录                     │    │
│  └─────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────┘
```

#### 2.2.1 分解引擎（Decomposition Engine）

负责将高层指令解析为子任务 DAG。核心工作流：

1. **指令解析**：接收 OmX 下发的工作流指令，提取任务目标、约束条件、优先级和上下文信息。
2. **上下文分析**：分析当前系统状态、Agent 可用性、文件锁状态、历史执行记录。
3. **分解策略选择**：根据任务类型选择合适的分解策略：
   - **按文件拆分**：大型代码库重构，按目录/文件拆分
   - **按功能拆分**：多模块开发，按功能领域拆分
   - **按阶段拆分**：流水线式任务，按处理阶段拆分
   - **按数据分片**：大数据处理，按数据分区拆分
4. **子任务生成**：生成 CC Task 格式的子任务列表，包含依赖关系定义。

#### 2.2.2 调度引擎（Scheduling Engine）

负责将子任务分配给合适的 Agent，并控制执行顺序：

1. **优先级队列**：维护一个多级优先级的就绪队列
2. **依赖解析**：利用 CC Task 的 `blockedBy` 字段构建依赖图，只有当所有前置任务完成时，任务才进入就绪队列
3. **Agent 选择**：基于以下策略选择 Agent：
   - 能力匹配（Agent 注册的能力集）
   - 负载均衡（当前运行任务数最少的优先）
   - 亲和性（同一上下文的后续任务优先分配给同一 Agent）
4. **资源预留**：在任务分配前检查文件锁和资源可用性

#### 2.2.3 归并引擎（Merge Engine）

负责将多个 Agent 的输出整合为最终结果：

1. **格式统一**：确保每个 Agent 的输出符合标准格式
2. **冲突检测**：检测多个 Agent 输出中的冲突（如修改同一文件的不同部分）
3. **合并策略**：
   - **顺序拼接**：按任务执行顺序拼接输出
   - **结构化合并**：按文件/模块结构合并，类似 Git Merge
   - **投票合并**：多个 Agent 执行相同任务，取多数结果
   - **分层合并**：按优先级层级合并，高优先级覆盖低优先级
4. **一致性校验**：验证合并结果的整体一致性

#### 2.2.4 状态管理（State Manager）

维护全局任务状态视图：

- 使用集中式状态存储（文件系统 + 内存缓存）
- 状态变更触发事件通知
- 支持持久化和恢复

#### 2.2.5 错误恢复（Error Recovery）

处理 Agent 执行过程中的异常：

- **超时处理**：任务超过预期时间 → 尝试取消 → 重新调度
- **失败重试**：可重试的任务自动重新分配（最多 N 次）
- **降级策略**：非关键任务失败 → 标记跳过，继续执行
- **回滚机制**：关键写操作失败 → 触发回滚
- **兄弟撤销**：借鉴 StreamingToolExecutor 的兄弟 abort 机制，一个 Agent 失败时通知同组其他 Agent 取消关联任务

## 3. 指令分解

### 3.1 分解粒度原则

任务分解的粒度直接影响系统效率：

| 粒度 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 过粗 | 上下文完整，通信少 | 并行度低，单 Agent 瓶颈 | 强顺序依赖的任务 |
| 适中 | 平衡并行和开销 | 需要精确设计 | 大多数场景 |
| 过细 | 理论并行度高 | 协调开销大，上下文碎片化 | 纯计算密集型 |

**经验法则**：每个子任务应在 2~15 分钟内完成，任务上下文不超过 3000 token。

### 3.2 分解流程详解

```
OmX 指令: "修复后端 API 中的认证漏洞并更新前端登录页面"
                        │
                        ▼
┌────────────────────────────────────────────┐
│              分解引擎                        │
│                                            │
│  Step 1: 语义解析                            │
│  ├─ 实体: [后端API, 认证漏洞, 前端登录页面]    │
│  ├─ 动作: [修复, 更新]                       │
│  └─ 领域: [backend_api, frontend]           │
│                                            │
│  Step 2: 能力匹配                            │
│  ├─ backend_api → Agent A (Rust/Node)       │
│  ├─ security_check → Agent B (审计工具)      │
│  └─ frontend_page → Agent C (React/TS)      │
│                                            │
│  Step 3: 依赖分析                            │
│  ├─ security_check (blockedBy: [])           │
│  ├─ backend_api_fix (blockedBy: [security])  │
│  └─ frontend_update (blockedBy: [backend])   │
│                                            │
│  Step 4: Task 生成                           │
│  ├─ task: security_audit, owner: agent_B    │
│  ├─ task: api_fix, blockedBy: [security]    │
│  └─ task: frontend_update, blockedBy: [api] │
└────────────────────────────────────────────┘
```

### 3.3 分解输出格式

每个子任务的输出使用以下结构（兼容 CC Task 格式）：

```yaml
task:
  id: "B-DEEP-042"
  name: "backend_api_fix"
  owner: "agent_A"
  description: "修复后端认证中间件的 JWT 验证漏洞"
  blockedBy:
    - "B-DEEP-041"   # security_audit
  input:
    scope: "src/middleware/auth.ts"
    context: "JWT token 验证缺失过期检查"
    files:
      - "src/middleware/auth.ts"
      - "src/utils/jwt.ts"
  success_criteria:
    - "所有 JWT token 必须验证过期时间"
    - "过期 token 返回 401 状态码"
    - "测试覆盖新增的验证逻辑"
  timeout: 300000    # 5 分钟
  retry: 2
  priority: 1
```

## 4. 结果归并

### 4.1 归并策略矩阵

根据任务类型选择不同的归并策略：

| 任务类型 | 归并策略 | 冲突处理 | 示例 |
|---------|---------|---------|------|
| 代码生成 | 结构化合并 | Git 风格三路合并 | 多模块生成 |
| 代码审查 | 投票合并 | 取多数意见 | 安全审计 |
| 文档编写 | 顺序拼接 | append 模式 | API 文档生成 |
| 数据分析 | 分层合并 | 优先级覆盖 | 报告生成 |
| 测试执行 | 聚合合并 | 取汇总结果 | 并行测试 |

### 4.2 结构化合并流程

针对代码生成场景的详细合并流程：

```
Agent A 输出                    Agent B 输出
  src/middleware/auth.ts          src/routes/login.ts
  src/utils/jwt.ts                src/routes/register.ts
  src/types/auth.ts               src/validators/auth.ts
        │                                │
        └────────────┬───────────────────┘
                     ▼
          ┌──────────────────────┐
          │    归并引擎           │
          │                      │
          │  1. 文件集合计算       │
          │     ├─ 无交集 → 直接合并
          │     ├─ 有交集 → 三路合并
          │     └─ 删除源文件 → 检查
          │                      │
          │  2. 依赖完整性检查     │
          │     ├─ import 路径有效
          │     ├─ 类型定义存在
          │     └─ 无循环依赖      │
          │                      │
          │  3. 风格统一           │
          │     ├─ 格式化对齐      │
          │     └─ 命名风格一致    │
          │                      │
          │  4. 输出包装           │
          │     ├─ 生成 diff 补丁  │
          │     └─ 写入目标目录    │
          └──────────────────────┘
```

### 4.3 冲突处理规则

当多个 Agent 修改同一文件的不同部分时：

1. **自动合并**：行级别冲突，若修改的是不同函数/块，自动合并
2. **标记冲突**：同一块被多个 Agent 修改，生成冲突标记交由人工裁决
3. **优先级覆盖**：高优先级 Agent 的修改覆盖低优先级
4. **时间戳优先**：较新的修改覆盖较旧的修改

## 5. 与 A-Line 的联动

### 5.1 协调器如何调用 A-Line 各子系统

B-Line 协调器与 A-Line 各子系统的交互关系：

```
┌──────────────────────────────────────────────────────┐
│                    B-Line Coordinator                  │
└──┬──────┬──────┬──────┬──────┬──────┬──────┬─────────┘
   │      │      │      │      │      │      │
   ▼      ▼      ▼      ▼      ▼      ▼      ▼
 A-01   A-02   A-03   A-04   A-05   A-06   A-07
 诊断    规划    修订   提示    测试    回滚    总结
```

#### 5.1.1 A-01 诊断系统

- 协调器将 Agent 执行的错误输出转发给 A-01 进行深度诊断
- 诊断结果作为任务重试的上下文补充

#### 5.1.2 A-02 规划系统

- 复杂任务先经 A-02 规划，再由协调器分解执行
- 规划结果用于指导 Agent 的执行顺序

#### 5.1.3 A-03 修订系统

- 使用 CC Task System 的修订流程管理 Agent 任务的生命周期
- 任务状态变更时触发 A-03 的修订检查

#### 5.1.4 A-04 提示工程

- 协调器从 A-04 获取 Agent 的 System Prompt 模板
- 根据任务类型选择或组合不同的提示模板

#### 5.1.5 A-05 测试框架

- 任务完成后的验证阶段，调用 A-05 的测试套件
- 测试结果影响归并决策

#### 5.1.6 A-06 回滚

- 当 Agent 执行需要回滚时，协调器调用 A-06 的 rollback 流程
- 回滚完成后，任务重新进入等待队列

#### 5.1.7 A-07 总结

- 所有子任务完成后，协调器调用 A-07 生成执行总结报告
- 报告存档用于后续优化

### 5.2 事件通知链

```
Coordinator 任务完成
    │
    ▼
事件发布器
    │
    ▼
clawhip 事件路由
    ├── Discord → 人类可读的执行报告
    ├── Git → 自动 commit + PR
    ├── CI → 触发测试流水线
    ├── 定时器 → 检查是否需要定期执行
    └── Agent 生命周期事件 → 通知下一阶段
```

### 5.3 错误传播与联动

当某个 Agent 执行失败时，协调器决定错误的传播范围：

- **局部错误**：仅影响当前子任务，重试或跳过
- **级联错误**：影响依赖链上的所有下游任务，协调器执行级联取消
- **全局错误**：影响整个工作流，协调器触发全流程回滚并通过 A-06 通知

## 6. 实现路径总结

### 6.1 实施阶段

| 阶段 | 内容 | 里程碑 |
|------|------|--------|
| Phase 1 | Coordinator 核心框架搭建：任务解析、分解、分配 | 单 Agent 协调工作 |
| Phase 2 | 多 Agent 并行调度：依赖图、就绪队列、兄弟撤销 | 双 Agent 并行执行 |
| Phase 3 | 结果归并引擎：结构化合并、冲突检测 | 多 Agent 输出自动归并 |
| Phase 4 | 事件路由集成：clawhip 对接 | 自动通知、CI 触发 |
| Phase 5 | OmX 工作流编排：定时触发、条件触发 | 全自动工作流 |

### 6.2 关键技术选择

- **任务模型**：CC Task 格式（兼容 A-Line/03）
- **Agent 通信**：MCP StdioClientTransport / WebSocketTransport
- **状态存储**：文件系统（JSON）+ 内存缓存
- **事件路由**：clawhip（兼容 Discord / Git / CI）
- **安全控制**：CC 权限模型（05-Agent-Permission-from-CC）
