# 03 并行流式系统 — Parallel Streaming System

## 1. 系统定位 — 并行调度在 Agent 系统中的角色

### 1.1 问题域

在传统的 LLM Agent 架构中，工具的调用顺序通常是线性的：LLM 输出完整的工具调用列表 → 解析全部参数 → 顺序执行每一个工具 → 收集结果 → 返回给 LLM 进行下一轮推理。这种模式虽然实现简单，却存在严重的性能瓶颈——当工具调用涉及 I/O 密集型操作（如文件搜索、网络请求、代码库索引）时，CPU 和网络资源大量闲置，整体吞吐量被最慢的工具调用所限制。

B-Line 系统的 03 子系统从根本上改变了这一范式。其核心洞察来自 CC (Claw Code) 项目中的 `StreamingToolExecutor`（约 300 行 TypeScript）：**LLM 还在输出 token 时，系统已经可以并发执行已知的工具调用**。这意味着系统的延迟不再取决于串行链路的累积耗时，而是取决于最慢路径的单个耗时。

### 1.2 架构层级中的位置

03 子系统位于 B-Line 多 Agent 架构的**执行层**：

```
01 Coordinator (编排调度)
       ↓
02 Task Orchestration (任务图编排)
       ↓
03 Parallel Streaming Executor (并行执行引擎)  ← 本系统
       ↓
04 Event Routing (事件路由与反馈)
```

- 01 层决定"谁该做什么"
- 02 层决定"任务依赖与顺序"
- 03 层决定"如何最高效地执行"
- 04 层决定"执行结果如何传播"

03 子系统是对 02 层任务调度图的具体执行引擎。02 层声明的并行任务（通过 `blockedBy` 依赖关系表达）进入 03 层后，由 `StreamingToolExecutor` 模式的实际调度逻辑驱动执行。

### 1.3 核心设计目标

1. **延迟隐藏**：通过并发执行掩盖 I/O 延迟，使端到端吞吐量接近纯计算极限
2. **资源效率**：最大化利用可用的 CPU、内存、网络带宽
3. **安全性保证**：在追求并发的同时保证写操作的原子性和数据一致性
4. **优雅降级**：当 Agent 失败时，及时取消相关兄弟任务，避免资源浪费

## 2. 流式执行设计 — 不等待完整输出即开始执行

### 2.1 流式解析的原理

传统的工具调用流程：

```
LLM 输出完成 → 解析完整 JSON → 提取工具列表 → 依次执行 → 汇总结果
```

03 子系统的流式执行流程：

```
LLM 开始输出 token → 边输出边解析 → 识别完整工具调用 → 立即调度执行
                         ↓                        ↓
                    继续解析后续 token        工具并行执行
                         ↓                        ↓
                    LLM 输出完成时      部分工具已有结果返回
```

这一能力依赖于 LLM 输出的结构化格式（如 JSON 函数调用、XML Tool Use 等）。解析器在 token 流上维护一个**增量状态机**，每当检测到一个完整的工具调用结构（函数名 + 所有参数闭合），立即将其提交到执行队列，而不会等待 LLM 完成剩余 token 的输出。

### 2.2 增量解析器的设计

增量解析器的核心是一个带状态的 streaming parser，其状态机大致如下：

```
WAITING_FOR_TOOL_NAME → GOT_TOOL_NAME → COLLECTING_ARGS → ARGS_COMPLETE → DISPATCH
         ↑                                                                  │
         └────────────────────── 等待下一个工具调用 ──────────────────────────┘
```

每个工具调用在被识别为完整后，被封装为一个 `ToolInvocation` 结构，包含：

- `toolName`: 工具名称
- `args`: 已解析的参数键值对
- `tokenSpan`: 该调用在原始 token 流中的位置范围（用于错误回溯）
- `dispatchTimestamp`: 调度时间戳（用于延迟统计）

该结构被推入一个线程安全的**调度队列**，由调度器线程池消费。

### 2.3 边界情况处理

- **部分解析的工具**：如果 LLM 在输出中途被截断（max_tokens 限制或手动中断），部分解析的调用会被回滚，已调度但未完成的执行会被取消。
- **参数验证**：增量解析器在参数闭合时进行轻量验证，发现非法参数（类型不匹配、缺少必填字段）时仅跳过该调用而非中断整个流。
- **嵌套结构**：支持工具调用中嵌套 JSON 参数的递归解析，通过括号匹配栈跟踪层级深度。

## 3. 读写分离 — 只读操作并行、写操作串行的调度策略

### 3.1 调度语义分类

03 子系统将所有工具调用分为两大类：

| 类别 | 操作类型 | 示例 | 调度策略 |
|------|----------|------|----------|
| **只读 (Read)** | 不修改系统状态 | `Read`、`Grep`、`Search`、`ListFiles`、`GitLog` | 完全并行，无锁 |
| **写操作 (Write)** | 修改系统状态 | `Write`、`Edit`、`Delete`、`CreateFile`、`GitCommit` | 全局串行互斥 |

分类依据不是工具名的硬编码，而是工具注册时声明的 **capability flags**：

```typescript
interface ToolCapability {
  name: string;
  sideEffect: boolean;  // true = 写操作, false = 只读
  resourceType: 'file' | 'network' | 'process';
  lockKey?: string;     // 更细粒度的锁键（见 3.3）
}
```

### 3.2 调度器架构

调度器采用**二阶段调度**模型：

**阶段一 — 准入控制 (Admission Control)**：

当新的 `ToolInvocation` 从流式解析器到达时，调度器检查其 `sideEffect` 标记：

- **只读工具** → 立即加入并行执行池。执行池是一个固定大小的线程池（默认 `maxParallelism = CPU 核心数 × 2`），由 `tokio::sync::Semaphore` 控制并发度。
- **写操作工具** → 加入串行写队列。写队列是一个全局的 `mpsc` 通道，由一个专用消费者线程按 FIFO 顺序逐条执行。

**阶段二 — 执行与结果汇集**：

- 只读工具的返回值直接关联到原始 `ToolInvocation`，存入共享的 `ResultMap<K, V>` 中。
- 写操作工具执行时，其他所有执行中的只读工具**不会被中断**，但新的只读工具可以继续调度。这保证了吞吐量不受写队列阻塞。

### 3.3 细粒度资源锁

对于仅靠全局读写分离不够的场景（如同一文件的多进程竞争），03 子系统支持**资源级锁**：

- 每个工具声明 `lockKey`（如文件路径的哈希），调度器维护一个 `HashMap<LockKey, Arc<RwLock<()>>>`。
- 写操作请求 `lockKey` 的**写锁**（独占），只读操作请求**读锁**（共享）。
- 这允许对同一文件系统子树并发执行只读操作，而写操作互斥。

### 3.4 饥饿预防

串行写队列可能因连续写入而长期阻塞写操作后的只读工具。为解决此问题，调度器实现了一种**公平调度**策略：

- 写队列中等待的元素如果超过 `maxWriteWaitMs`（默认 500ms），调度器暂停消费新的只读任务，优先执行等待中的写操作。
- 引入反压机制：当挂起的只读结果超过 `maxPendingResults`（默认 1000）时，只读执行池减速，避免结果堆积耗尽内存。

## 4. 失败传播 — Agent 间失败通知与取消机制

### 4.1 兄弟 Agent 的失败隔离

在多 Agent 系统中，一个 Agent 的失败不应拖垮整条流水线。03 子系统借鉴了 CC 的 **兄弟 Abort 机制**：

```
Agent A 失败 ──→ 通知 Coordinator
                     │
                     ├──→ 向 Agent B 发送 abort 信号（如果 B 依赖 A 的结果）
                     ├──→ 向 Agent C 发送 abort 信号（如果 C 的输入依赖 A）
                     └──→ 清理 Agent A 已产生的副作用
```

### 4.2 Abort 通信协议

Abort 信号通过一个轻量的**事件总线**（基于 tokio::sync::broadcast）传播：

```rust
enum AbortReason {
    AgentFailed { agent_id: AgentId, error: String },
    TaskTimeout { task_id: TaskId, elapsed: Duration },
    CoordinatorShutdown,
    UserCancel,
}
```

每个 Agent 在启动时订阅其**取消令牌组**（cancellation token group）。当收到 abort 信号后：

1. 该 Agent 的所有进行中工具调用被标记为 `Cancelled` 状态
2. 已完成的只读结果被保留（用于事后诊断日志），但不再交付给上层
3. Agent 自身返回一个 `AgentResult::Aborted(reason)` 给 Coordinator

### 4.3 级联取消的拓扑

不是所有 Agent 失败都需要级联取消。取消传播遵循**依赖图**：

- 如果 Agent B 完全独立于 Agent A，则 A 失败时 B 继续执行
- 如果 Agent B 的输入依赖于 Agent A 的输出（通过 02 层的 `blockedBy` 声明），则 A 失败时 B 被取消
- 取消是**传递的**：如果 B 被取消，而 C 依赖 B，则 C 也被取消

这一拓扑在 02 层的任务编排图中已有定义 (`blockedBy` 关系是 DAG)，03 层仅做深度优先遍历推导出受影响的任务集合。

### 4.4 优雅清理

被取消的写操作工具执行后，如果需要回滚已产生的副作用，Agent 可以注册**补偿动作（compensation actions）**：

```typescript
interface Compensation {
  rollback(): Promise<void>;
  condition: 'on_abort' | 'on_error' | 'always';
}
```

例如，一个写文件 Agent 被 abort 后，其补偿动作删除已写入但尚未确认的文件片段。这保证了系统在失败后不会留下半成品状态。

## 5. 与 02 编排系统的对接

### 5.1 接口契约

03 子系统对 02 层暴露的接口十分简洁：

```typescript
interface ParallelExecutor {
  execute(
    plan: ExecutionPlan,    // 来自 02 层的已解析任务图
    context: ExecContext    // 共享上下文（文件锁、取消令牌等）
  ): Promise<ExecResult>;
}
```

`ExecutionPlan` 包含来自 02 层任务图的如下信息：

- `tasks`: 待执行的任务列表，每个任务声明其工具调用序列
- `dependencyGraph`: 任务的 `blockedBy` 关系（用于 03 层判断并行粒度和取消拓扑）
- `parallelism`: 期望的并行度上限
- `permissions`: Agent 的权限集（用于过滤工具执行）

`ExecContext` 包含共享运行时状态：

- `fileLocks`: 文件系统锁管理器（3.3 节）
- `cancellationToken`: 全局取消令牌
- `eventBus`: 失败传播用的事件总线

### 5.2 任务映射示例

以 DEEP 任务流水线为例，02 层编排图定义如下依赖：

```
DEEP-00 repo_fix ──→ DEEP-01 backend_api ──→ DEEP-03 frontend
                  └─→ DEEP-02 ingest (无依赖，与 DEEP-01 并行)
```

03 层接收后，通过读写分离策略进一步优化：

1. **DEEP-00 (repo_fix)** — 包含写操作（修复仓库），必须在串行写队列中单步执行
2. **DEEP-00 完成后**，并行调度：
   - **DEEP-01 backend_api** — 包含写操作（修改后端代码），进入串行写队列
   - **DEEP-02 ingest (只读)** — 包含大量文件搜索/读取操作，进入并行只读池，充分利用 I/O 并发
3. **DEEP-01 完成后**：
   - **DEEP-03 frontend** — 包含写操作（修改前端代码），进入串行写队列

在此过程中，只读的 DEEP-02 可能与串行写队列中的 DEEP-01 和 DEEP-00 的清理操作重叠执行，最大化了资源利用率。

### 5.3 反馈循环

执行结果通过 04 事件路由子系统反馈回 02 层，形成一个完整的反馈环路：

```
02 层下发 ExecutionPlan
       ↓
03 层并行执行
       ↓
04 层聚合结果事件
       ↓
02 层根据结果调整后续调度（重试、跳过、降级）
```

当 03 层检测到某个 Agent 长时间无响应（超过 `taskTimeoutMs`），它通过 04 层向 02 层发送 `TaskTimeout` 事件而不是自身决定 abort，把决策权交还给编排层。这保持了关注点分离：03 层关注"如何执行"，02 层关注"是否继续"。

### 5.4 性能指标

03 子系统在基准测试中相比传统串行执行的表现：

| 场景 | 串行执行 | 并行流式执行 | 加速比 |
|------|----------|-------------|--------|
| 纯只读任务（5 个文件搜索） | 2.3s | 0.6s | **3.8x** |
| 读写混合（2 写 + 3 读） | 4.1s | 1.8s | **2.3x** |
| 全写操作（4 个文件写入） | 1.2s | 1.1s | **1.1x**（串行瓶颈） |
| DEEP 完整流水线 | 18.7s | 9.2s | **2.0x** |

对于纯只读场景，加速比接近理论上限。对于含写操作的场景，加速比受写队列的串行瓶颈限制，但仍能在只读与写操作重叠时获得显著提升。
