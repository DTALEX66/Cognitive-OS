# 04 事件路由系统 — Event Routing System

## 1. 系统定位 — 事件路由的角色

### 1.1 从手动触发到事件驱动

在 B-Line 系统之前的版本（A-Line）中，Agent 的启动完全依赖人工操作：开发者需要手动在 CODEX 中输入指令、等待 DEEP 完成、检查结果、再输入下一个指令。这种模式不仅效率低下，而且使得系统无法响应外部事件（如 git push、新 Issue、定时触发的维护任务）。

04 子系统——事件路由系统，其核心角色是**将 B-Line 从被手动调用的工具转变为主动响应事件的自治系统**。它源自 CW（Claw Code）项目中的 clawhip 组件——一个轻量级的事件路由守护进程，能够监控多种事件源并根据配置规则自动触发动作。

参考来源：https://github.com/Yeachan-Heo/clawhip

### 1.2 系统边界

04 子系统在 B-Line 架构中的位置：

```
01 Coordinator（编排决策）         05 Agent Permission（安全边界）
       ↕                                ↕
02 Task Orchestration（任务图）    06 MCP Extension（工具扩展）
       ↕
03 Parallel Streaming（并行执行）
       ↕
04 Event Routing（事件路由）-- 本系统
       ↕
外部世界（git, Discord, Issues, CI, Timer...）
```

- **向上**：04 层接收 03 层的执行结果事件，将其格式化并路由到外部输出通道（Discord 通知、git commit 消息、CI 构建触发）
- **向下**：04 层监听外部事件源，将其标准化为内部事件，路由到 01 Coordinator 启动新的工作流
- **横向**：04 层与 05 权限系统交互——入站事件需要经过权限验证才能触发动作

### 1.3 核心价值

| 维度 | 手动模式 | 事件驱动模式 |
|------|----------|-------------|
| 响应延迟 | 取决于人看到消息 + 输入指令 | 毫秒级自动响应 |
| 覆盖时段 | 工作时间 | 7×24 |
| 可重复性 | 因人而异 | 严格一致 |
| 可审计性 | 依靠个人记录 | 完整事件日志 |
| 规模化 | 一人同时处理一个任务 | 同时处理多路事件流 |

## 2. 事件源设计 — 可接入的事件类型

### 2.1 标准化事件模型

所有事件源在被 04 系统接收后，被归一化为统一的事件结构：

```rust
struct Event {
    id: EventId,                         // 全局唯一 ID（ULID）
    source: EventSource,                 // 来源标识
    event_type: EventType,               // 事件类型枚举
    timestamp: DateTime<Utc>,
    payload: HashMap<String, Value>,     // 结构化负载
    severity: Severity,                  // Info / Warning / Error / Critical
    correlation_id: Option<String>,      // 关联 ID（用于追踪同一次级联）
}
```

其中 EventSource 是一个代数类型：

```rust
enum EventSource {
    Git { repo: String, remote: Url, branch: String },
    Tmux { session: String, window: String },
    GitHub { owner: String, repo: String, issue_number: Option<u64> },
    Discord { channel_id: String, guild_id: String },
    Agent { agent_id: AgentId, state: AgentState },
    Timer { cron: String, name: String },
    Webhook { url: String, method: HttpMethod },
    FileSystem { path: PathBuf, change_type: FsChange },
}
```

### 2.2 git 事件源

监控本地和远程 git 仓库的变化：

- **git push 事件**：检测到本地推送完成或远程仓库有新提交后触发。包含提交范围、提交信息、作者、分支名。
- **git tag 事件**：新标签创建时触发，常用于版本发布流水线。
- **git merge 事件**：PR 合入目标分支时触发，触发回归测试和部署。
- **实现方式**：通过 git hooks（post-receive, post-commit, post-merge）或周期性 git fetch + 轮询。

### 2.3 tmux 会话事件源

监控 tmux 终端会话的生命周期和输出模式：

- **会话创建/销毁**：Agent 或人新建或关闭 tmux 会话时触发。
- **输出匹配**：在 tmux 窗格的输出流上执行正则匹配，匹配到关键字时触发对应事件。
- **实现方式**：通过 tmux 的 capture-pane + hooks API，以轮询间隔检查窗格内容变化。

### 2.4 GitHub Issues / PR 事件源

通过 GitHub API 或 Webhook 接收平台事件：

- **Issue 创建**：新 Issue 被创建时触发代码分析或自动化回复。
- **Issue 评论**：特定用户或包含特定关键词的评论触发任务分配。
- **PR 提交/合入**：PR 创建时触发表单验证或自动 reviewer 分配；PR 合入时触发部署流水线。
- **实现方式**：通过 GitHub Webhook 直接接收（推荐）或通过 GitHub API 轮询（备选）。

### 2.5 Discord 事件源

作为人机接口的主要通道：

- **频道消息**：在指定频道中收到消息时触发。支持按用户、角色、关键词过滤。
- **Slash 命令**：Discord 的斜杠命令直接映射为系统命令。
- **Thread 创建**：Discord 论坛频道的新 thread 触发新的任务链。
- **实现方式**：通过 Discord Gateway Intents 接收实时事件流。

### 2.6 Agent 生命周期事件源

Agent 自身也是事件源（内反馈）：

- **Agent 启动/终止**：Agent 上线或下线时触发资源分配或清理逻辑。
- **Agent 任务完成**：Agent 完成一个 Task 时，将结果打包为事件，触发下游 action。
- **Agent 异常**：Agent 抛出未处理异常或超时时，触发告警和恢复流程。
- **心跳超时**：如果 Agent 在 heartbeatInterval 内未发送心跳，触发故障转移事件。

### 2.7 定时器事件源

基于 cron 表达式的周期性触发：

- 每日学习诊断：每日凌晨触发 DEEP 学习流水线
- 每周总结：每周一触发仓库状态报告生成
- 维护窗口：每月初触发依赖更新和清理任务
- 健康检查：每 5 分钟触发系统组件存活检查

## 3. 路由规则 — 事件到动作的映射

### 3.1 规则定义格式

路由规则以声明式 YAML 定义：

```yaml
rules:
  - id: "git-push-test"
    description: "git push 后自动运行测试流水线"
    match:
      source: Git
      event_type: Push
      condition: event.payload.branch == "main"
    actions:
      - invoke: coordinator.start_workflow
        args:
          workflow: "test-pipeline"
          params:
            commit: event.payload.commit_hash
      - notify:
          channel: Discord
          target: "build-notifications"
          template: "测试流水线已启动"
```

### 3.2 匹配引擎

路由规则匹配引擎是一个轻量的 Retes 算法（forward chaining）实现：

1. **事件标准化**：原始事件被归一化为 Event 结构
2. **规则索引**：所有规则按 (source, event_type) 建立哈希索引，O(1) 定位候选规则
3. **条件求值**：对候选规则逐一求值 condition 表达式

条件表达式支持字段访问、比较操作、逻辑组合和模板变量插值。

### 3.3 Action 类型

匹配规则后触发的 action 类型：

| Action | 含义 | 示例 |
|--------|------|------|
| invoke | 调用内部系统功能 | 启动 Coordinator 工作流 |
| notify | 发送通知到外部通道 | Discord 消息、邮件 |
| transform | 转换事件并重新路由 | Agent 完成事件转为通知 |
| webhook | 调用外部 HTTP API | 触发 CI 构建 |
| delay | 延迟后执行子动作 | 等待后检查构建结果 |
| suppress | 抑制匹配事件（去重） | 同一 commit 只触发一次 |

### 3.4 去重与幂等性

为防止同一事件触发多次，04 层实现三种去重策略：

1. **事件 ID 去重**：基于 event.id 的布隆过滤器，同一 ID 的事件只处理一次
2. **幂等键去重**：用户可指定 idempotency_key 表达式，在 idempotency_window 内相同键的事件被丢弃
3. **滑动窗口去重**：对于高频同类事件，在一个时间窗口内合并为单一事件

## 4. 自动触发 — 无需人工介入的自动化流程

### 4.1 完整自动化链路示例

以 git push 到 main 分支事件为例：

```
Step 1:  git post-receive hook 触发文件系统写事件
Step 2:  04 检测到 git push，标准化为 Event
Step 3:  路由规则匹配，git-push-test 规则命中
Step 4:  action 触发 coordinator.start_workflow
Step 5:  01 Coordinator 解析流水线，生成 ExecutionPlan
Step 6:  02 Task Orchestration 编排任务图
Step 7:  03 Parallel Executor 并行执行测试任务
Step 8:  测试完成，Agent 生命周期事件
Step 9:  04 收到 Agent 完成事件，路由规则匹配
Step 10: action 通知 Discord 构建频道
Step 11: 如果测试失败，webhook 自动创建 Issue
```

全程无需人工介入。

### 4.2 定时自动化示例

```yaml
- id: "daily-learning-diagnostic"
  match:
    source: Timer
    event_type: Cron
    condition: event.payload.cron == "0 3 * * *"
  actions:
    - invoke: coordinator.start_workflow
      args:
        workflow: "learning-diagnostic"
```

### 4.3 安全护栏

自动触发不能无限制执行。04 层内置以下安全机制：

- **速率限制（Rate Limiting）**：每个规则每分钟最多触发 N 次，超限事件进入死信队列。
- **断路器（Circuit Breaker）**：如果一个 action 连续失败超过 failureThreshold，该规则自动熔断。
- **权限校验**：每个 action 执行前调用 05 权限系统验证权限对。
- **审计日志**：所有触发动作记录到 event_audit_log。

## 5. 与 01-03 子系统的联动

### 5.1 与 01 Coordinator 的联动

04 层是 01 Coordinator 的外部触发器：

- 04 标准化外部事件，匹配路由规则后通过 invoke 触发 01 解析工作流定义
- 01 分解为步骤、生成 ExecutionPlan、下发到 02 层、返回 WorkflowId
- 01 的工作流执行完毕时通知 04 记录

关键接口：

```rust
trait WorkflowApi {
    async fn start_workflow(workflow: String, params: HashMap<String, Value>) -> Result<WorkflowId>;
    async fn get_workflow_status(id: WorkflowId) -> Result<WorkflowStatus>;
    async fn cancel_workflow(id: WorkflowId) -> Result<()>;
}

trait EventBus {
    async fn publish(event: Event) -> Result<()>;
    async fn subscribe(pattern: EventPattern) -> Receiver<Event>;
}
```

### 5.2 与 02 Task Orchestration 的联动

04 层的事件经过 01 转换后，最终到达 02 层的任务编排。02 层的任务状态变更也作为事件反馈回 04 层：

- 02 层任务完成触发 TaskCompleted 事件，04 路由规则触发下一阶段或通知
- 02 层任务失败触发 TaskFailed 事件，04 路由规则触发降级策略或通知
- 02 层阻塞触发 TaskBlocked 事件，04 路由规则在超时后触发 escalation

### 5.3 与 03 Parallel Stream 的联动

03 层的执行状态直接产生事件，通过 04 层可以路由到实时监控仪表板、自动伸缩决策和诊断日志。

03 层产生的事件类型：

```rust
enum ExecutorEvent {
    ToolStarted { tool: String, agent: AgentId },
    ToolCompleted { tool: String, agent: AgentId, duration: Duration },
    ToolFailed { tool: String, agent: AgentId, error: String },
    ExecutorThrottled { pending_count: u32 },
    ExecutorIdle,
}
```

### 5.4 事件追踪与可观测性

04 层为所有子系统提供了跨子系统的事件追踪能力：每个进入 04 层的事件被分配 correlation_id，由该事件触发的所有工作流、任务、工具调用都传播同一 correlation_id。通过 correlation_id 可以在日志系统中追溯完整的端到端链路。

### 5.5 与 05 Agent Permission 的安全集成

每个入站事件在触发 action 前经过 05 权限系统的三层检查：

1. **事件源认证**：验证事件来源的真实性（git hook 签名验证、Webhook HMAC 校验）
2. **规则执行授权**：检查该事件源是否有权触发该规则
3. **Action 权限**：执行 action 时检查当前权限上下文是否允许该操作

这三层检查构成了纵深防御体系。

### 5.6 与 06 MCP Extension 的联动

04 层作为事件总线，可以接入通过 06 MCP Extension 注册的自定义事件源：

- MCP 插件可以注册自定义事件源（如 CloudWatchAlarm、自定义 Webhook）
- MCP 插件可以注册自定义 action handler（如 deploy_to_k8s、run_db_migration）
- 04 层的路由规则对自定义事件源和原生事件源一视同仁，统一处理

这保证了 B-Line 的事件驱动架构可以通过 MCP 生态持续扩展。
