## 1. 系统定位

### 1.1 任务编排在多 Agent 系统中的角色

任务编排系统是 B-Line 中承上启下的**调度中枢**。如果说 01 协调器系统（Coordinator）是大脑，负责"做什么"和"谁来做"，那么任务编排系统就是神经网络，负责"什么时候做"和"按什么顺序做"。

任务编排系统的核心职责包括：

- **任务建模**：将 DEEP-01~99 的任务流水线映射为结构化的 CC Task 对象，每个任务携带完整的元数据、输入输出规范、依赖关系和执行约束。
- **依赖图管理**：构建并维护任务间的依赖关系有向图（DAG），确保每个任务在其所有前置条件满足后才进入就绪状态。
- **并行调度**：识别无依赖关系的任务，驱动多个 Agent 并行执行以最大化吞吐量。
- **资源仲裁**：管理文件锁、Agent 槽位、并发限制等共享资源的分配与释放。
- **执行保障**：提供任务超时、重试、死锁检测、级联取消等可靠性机制。
- **与协调器对接**：作为协调器的执行引擎，接收分解后的子任务，负责调度执行，并将结果回传给归并引擎。

### 1.2 从 CC Task 到 B-Line 的映射关系

CC Task 系统（来自 `cc-code/src/utils/tasks.ts`，约 779 行）是一个功能完备的任务调度框架，B-Line 任务编排直接继承其核心设计，并根据多 Agent 场景进行扩展。

```
CC Task 系统                              B-Line 任务编排
┌─────────────────┐                    ┌──────────────────────┐
│ task: id         │ ──────────────→  │ task: B-DEEP-{NN}     │
│ task: owner      │ ──────────────→  │ task: owner (Agent)   │
│ task: blockedBy  │ ──────────────→  │ task: blockedBy (DAG) │
│ task: status     │ ──────────────→  │ task: status (扩展)   │
│ task: metadata   │ ──────────────→  │ task: metadata + ctx  │

│ claim mechanism  │ ──────────────→  │ Agent 竞争/分配机制    │
└─────────────────┘                    └──────────────────────┘
```

## 2. 任务模型

### 2.1 CC Task 核心数据结构

B-Line 的任务模型直接继承 CC Task 的核心字段，并根据多 Agent 场景进行扩展：

```typescript
interface BLineTask {
  // === CC Task 兼容字段 ===
  id: string;                    // 全局唯一任务 ID，格式 "B-DEEP-{NNN}"
  name: string;                  // 人类可读的任务名称
  owner: string | null;          // 执行此任务的 Agent ID (null 表示未分配)
  blockedBy: string[];           // 依赖的前置任务 ID 列表
  status: TaskStatus;            // 当前执行状态

  // === B-Line 扩展字段 ===
  priority: number;              // 优先级 (0=critical, 1=high, 2=normal, 3=low)
  type: TaskType;                // 任务类型 (read_only / write / mixed)
  timeout: number;               // 超时时间 (毫秒)
  retryCount: number;            // 已重试次数
  maxRetries: number;            // 最大重试次数
  description: string;           // 详细任务描述
  input: TaskInput;              // 输入上下文
  output: TaskOutput | null;     // 执行输出
  successCriteria: string[];     // 成功判定标准
  createdAt: number;             // 创建时间戳
  startedAt: number | null;      // 开始执行时间戳
  completedAt: number | null;    // 完成时间戳
  agentCapability: string[];     // 要求的 Agent 能力标签
  fileLocks: string[];           // 需要占用的文件锁路径
  context: Record<string, any>;  // 自由格式上下文
  tags: string[];                // 标签 (用于分组和过滤)
}
```

### 2.2 任务状态机

任务在其生命周期中经历以下状态转换：

```
                   ┌──────────┐
                   │  PENDING  │
                   └────┬─────┘
                        │ 依赖满足
                        ▼
                   ┌──────────┐
        ┌────────→│  READY    │←─────────┐
        │         └────┬─────┘           │
        │              │ 分配 Agent       │ 重试
        │              ▼                 │
        │         ┌──────────┐           │
        │         │ RUNNING  │───────────┤
        │         └────┬─────┘  失败可重试│
        │              │                 │
        │     ┌────────┼────────┐        │
        │     ▼        ▼        ▼       │
        │ ┌──────┐ ┌──────┐ ┌──────┐    │
        │ │DONE  │ │FAILED│ │CANCELLED │
        │ └──────┘ └──────┘ └──────┘    │
        │                               │
        └───────────────────────────────┘
                 FAILED → 判定可重试 → READY
```

**状态定义**：

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| PENDING | 已创建，等待依赖满足 | 任务创建时 |
| READY | 依赖已满足，等待 Agent 分配 | 所有 blockedBy 任务状态为 DONE |
| RUNNING | 正在被 Agent 执行 | Agent 认领任务时 |
| DONE | 执行成功 | Agent 报告成功，通过校验 |
| FAILED | 执行失败 | Agent 报告失败或超时 |
| CANCELLED | 被取消 | 兄弟取消 / 级联取消 / 手动取消 |

### 2.3 任务生命周期详细流程

```text
时间线：
task.create()                     → status: PENDING
     ↓
依赖检查 (所有blockedBy已完成)     → status: READY
     ↓
调度器选择就绪任务                  → 放入 READY 队列
     ↓
Agent 认领 (claim)                → status: RUNNING, owner: agent_X
     ├─ 获取文件锁
     ├─ 加载输入上下文
     ├─ 执行 (Agent 独立工作)
     ├─ 释放文件锁
     └─ 报告结果
     ↓
结果验证 (校验 successCriteria)    → status: DONE / FAILED
     ↓
检查是否有下游任务变为 READY         → 重复调度
```

## 3. 依赖图

### 3.1 依赖图建模

依赖图的核心数据结构是一个**有向无环图（DAG）**，其中节点是任务，边表示依赖关系。

```
依赖图示例（DEEP-00 ~ DEEP-06）：

                    ┌─────────┐
                    │ DEEP-00 │
                    │repo_fix │
                    └────┬────┘
                         │
           ┌─────────────┼─────────────┐
           │             │             │
           ▼             ▼             ▼
     ┌──────────┐ ┌──────────┐   ┌──────────┐
     │ DEEP-01  │ │ DEEP-02  │   │ DEEP-07  │
     │backend_api│ │ ingest   │   │ (扩展)   │
     └─────┬────┘ └─────┬────┘   └──────────┘
           │             │
           │             │
           ▼             ▼
     ┌──────────┐ ┌──────────┐
     │ DEEP-03  │ │ DEEP-04  │
     │ frontend │ │ testing  │
     └─────┬────┘ └─────┬────┘
           │             │
           └──────┬──────┘
                  ▼
            ┌──────────┐
            │ DEEP-05  │
            │deploy    │
            └──────────┘

            ┌──────────┐
            │ DEEP-06  │
            │document  │
            └──────────┘
```

**并行性分析**：
- DEEP-01 和 DEEP-02 可以**并行执行**（都只依赖 DEEP-00）
- DEEP-03 必须等待 DEEP-01 完成后才能开始
- DEEP-04 等待 DEEP-01 和 DEEP-02 都完成后才能开始
- DEEP-03 和 DEEP-04 可以并行
- DEEP-06 只依赖 DEEP-01 和 DEEP-03，无其他依赖时可提前并行
- DEEP-05 是所有任务的汇合点

### 3.2 依赖图构建算法

```python
def build_dependency_graph(tasks: list[BLineTask]) -> DependencyGraph:
    graph = DependencyGraph()

    for task in tasks:
        graph.add_node(task.id, task)

        # 将 blockedBy ID 转换为依赖边
        for dep_id in task.blockedBy:
            if dep_id in task_map:
                graph.add_edge(dep_id → task.id)
            else:
                # 外部依赖标记
                graph.add_external_dep(task.id, dep_id)

    # 环检测
    cycles = graph.detect_cycles()
    if cycles:
        raise CyclicDependencyError(cycles)

    # 计算层级和关键路径
    for level in graph.topological_sort():
        for task_id in level:
            tasks[task_id].level = graph.level(task_id)

    return graph
```

### 3.3 关键路径分析

关键路径决定整个工作流的最短完成时间：

```
关键路径：DEEP-00 → DEEP-01 → DEEP-03 → DEEP-05
预估最短时间：2 + 10 + 8 + 3 = 23 分钟

如果 DEEP-01 执行时间超过预期，整个关键路径都会受影响。
调度器应优先保障关键路径上的任务资源。
```

**关键路径调度策略**：
1. 关键路径上的任务优先分配资源
2. 关键路径任务限制重试次数（避免过长时间的阻塞）
3. 非关键路径任务的延迟可以容忍，但不超过总松弛时间

### 3.4 动态依赖更新

在任务执行过程中，依赖图**理论上不可变**——但以下机制允许有限动态调整：

- **可选依赖**：标记为 `optional: true` 的依赖，即使前置失败也可执行
- **条件分支**：根据前一任务的输出决定是否添加新的依赖
- **运行时注册**：Agent 在执行过程中发现新的子任务依赖，可向编排器注册

## 4. 并行调度

### 4.1 并行调度算法

并行调度引擎核心是**事件驱动的就绪队列**模型：

```
调度循环：

1. 初始状态：所有 PENDING 任务入图

2. 事件循环：
   while 有未完成的任务:
     // 依赖满足检查
     ready_tasks = check_ready_tasks(graph)

     // 优先级排序
     ready_tasks.sort(key=priority, reverse=True)

     // 资源检查
     for task in ready_tasks:
       if available_agents() > 0:
         if acquire_file_locks(task.fileLocks):
           assign_task_to_agent(task)
           task.status = RUNNING

     // 等待事件（任务完成 / 超时 / 失败）
     event = wait_for_event()

     // 事件处理
     if event is TaskCompleted:
       release_file_locks(event.task)
       update_downstream_tasks(event.task)
     elif event is TaskFailed:
       if event.task.retryCount < event.task.maxRetries:
         event.task.retryCount++
         event.task.status = READY  # 重新入队
       else:
         handle_max_retry_exceeded(event.task)
     elif event is TaskTimeout:
       cancel_task(event.task)
       handle_task_timeout(event.task)
```

### 4.2 读写互斥规则

借鉴 StreamingToolExecutor 的读写分离策略，B-Line 任务按照操作类型执行不同的并发规则：

| 当前任务 | 已有只读任务 | 已有写任务 |
|---------|------------|-----------|
| 只读任务 | 可并行 ✅ | 阻塞 ⏳ |
| 写任务   | 阻塞 ⏳ | 阻塞 ⏳ |

**实现方式**：文件级读写锁

```
// 文件锁管理器
class FileLockManager {
    // 每个文件路径维护读锁计数器和写锁持有者
    locks: Map<string, { readCount: number; writeOwner: string | null }>;

    acquireReadLock(taskId: string, files: string[]): boolean {
        for (file of files) {
            if (this.locks[file]?.writeOwner !== null) {
                return false;  // 被写锁占用
            }
            this.locks[file] ??= { readCount: 0, writeOwner: null };
            this.locks[file].readCount++;
        }
        return true;
    }

    acquireWriteLock(taskId: string, files: string[]): boolean {
        for (file of files) {
            const lock = this.locks[file];
            if (lock && (lock.readCount > 0 || lock.writeOwner !== null)) {
                return false;  // 被任何锁占用
            }
            this.locks[file] = { readCount: 0, writeOwner: taskId };
        }
        return true;
    }
}
```

### 4.3 Agent 分配策略

| 策略 | 描述 | 适用场景 |
|------|------|---------|
| 轮询 (Round-Robin) | 按顺序循环分配 | 同质化任务池 |
| 最少负载 (Least-Loaded) | 分配给当前任务最少的 Agent | 异构任务，各任务耗时差异大 |
| 能力匹配 (Capability-Match) | 根据 Agent 注册的能力集匹配 | 需要特定工具链的任务 |
| 亲和性 (Affinity) | 尽量分配同一 Agent 给相关任务 | 减少上下文切换开销 |
| 抢占 (Preemptive) | 高优先级任务可抢占低优先级 Agent | 紧急修复场景 |

**默认策略组合**：能力匹配 → 亲和性 → 最少负载

### 4.4 并发度控制

为防止系统资源过载，编排器实施多级并发控制：

```
并发度配置:
├── 全局最大 Agent 数: MAX_AGENTS = 5
├── 每个 Agent 最大任务数: MAX_TASKS_PER_AGENT = 1
├── 写操作最大并行数: MAX_WRITE_PARALLEL = 1
├── 同一文件最大并行修改者: 1
└── 同一命名空间最大并行任务: 3
```

**动态调整**：系统可根据当前 Agent 响应时间、任务排队长度和资源利用率，动态调整并发度上限。

### 4.5 死锁预防

依赖图 + 文件锁的双重锁定场景下，需要死锁预防机制：

1. **全局顺序加锁**：所有任务按文件路径字典序获取锁
2. **锁超时**：等待文件锁超过 30 秒 → 放弃并重新排队
3. **死锁检测**：周期性运行 Wait-For Graph 检测
4. **超时中断**：任务总执行时间超过预期 → 中断并释放所有锁

### 5.4 错误传播与协调

当编排器遇到不可恢复的错误时，它与协调器的交互如下：

| 错误类型 | 编排器行为 | 协调器响应 |
|---------|-----------|-----------|
| Agent 不可用 | 重新排队，标记 owner 悬挂 | 尝试分配备用 Agent |
| 文件锁死锁 | 解锁所有被占文件，重试队列 | 无需特殊处理 |
| 依赖图环检测 | 初始化时抛出 CyclicDependencyError | 要求分解引擎重新分解 |
| 关键路径超时 | 暂停非关键路径任务，集中资源 | 评估是否需要缩短任务范围 |
| Agent 输出异常 | 标记任务 FAILED，触发重试逻辑 | 归并引擎标记该任务输出无效 |

### 5.5 完整集成示例

以下是完整的端到端集成流程，以"修复后端认证"任务为例：

```yaml
# Coordinates between 01-Coordinator and 02-Orchestrator

# Step 1: 协调器分解任务
coordinator:
  decomposition:
    - task: "B-DEEP-041", name: "security_audit",   type: "read_only"
    - task: "B-DEEP-042", name: "api_fix",          type: "write"
    - task: "B-DEEP-043", name: "frontend_update",  type: "write"

# Step 2: 协调器 → 编排器
coordinator → orchestrator:
  action: "submitTasks"
  tasks:
    - id: "B-DEEP-041", blockedBy: []
    - id: "B-DEEP-042", blockedBy: ["B-DEEP-041"]
    - id: "B-DEEP-043", blockedBy: ["B-DEEP-042"]

# Step 3: 编排器调度
orchestrator:
  scheduling:
    - B-DEEP-041 → READY → Agent_A (read_only, 可并行)
    - B-DEEP-042 → blocked, waiting for B-DEEP-041
    - B-DEEP-043 → blocked, waiting for B-DEEP-042

# Step 4: 编排器 → 协调器 (结果返回)
orchestrator → coordinator:
  results:
    - task: "B-DEEP-041", status: "DONE",   output: { vulnerabilities: [...] }
    - task: "B-DEEP-042", status: "DONE",   output: { patches: [...] }
    - task: "B-DEEP-043", status: "DONE",   output: { ui_changes: [...] }

# Step 5: 协调器归并
```

## 6. 可靠性保障

### 6.1 幂等性

所有任务设计为**幂等**（多次执行结果一致）。实现方式：

- Agent 执行前检查目标文件状态（是否已应用变更）
- 写操作先创建备份，失败时可恢复
- 任务结果可重复提交，编排器去重

### 6.2 持久化

编排器的状态持久化到文件系统：

```
.b-line/orchestrator/
├── state.json              # 当前所有任务状态快照
├── tasks/
│   ├── B-DEEP-041.json     # 单个任务的完整状态和上下文
│   ├── B-DEEP-042.json
│   └── ...
├── graph.json              # 依赖图序列化
└── logs/
    ├── schedule.log        # 调度事件日志
    └── execution.log       # 执行详细日志
```

### 6.3 恢复流程

```text
编排器崩溃恢复流程：

1. 重启后加载 state.json → 重建内存状态
2. 遍历所有 RUNNING 任务 → 状态标记为 UNKNOWN
3. 查询 Agent 执行状态：
   - Agent 仍在运行 → 标记为 RUNNING
   - Agent 已完成 → 标记为 DONE
   - Agent 无响应 → 标记为 FAILED（重试）
4. 重建依赖图 → 重新计算就绪任务
5. 继续调度循环
```
