## 1. Paper Rules

### Rule 1: NO LLM-generated paper summaries
LLM cannot replace academic databases. Every summary must come from a real, verifiable publication.

### Rule 2: No validation = no entry
Before entry, verify all four through CrossRef API:
- Title (exact match)
- First author surname
- Journal name
- Publication year

### Rule 3: Chinese papers must come from real databases
CNKI, Wanfang, COVID. No LLM-created Chinese papers allowed.

### Rule 4: Don't borrow real author names for fake papers
Creating non-existent papers under real researchers' names damages credibility.

### Rule 5: Complete traceable citation format required
Full author list, year, full journal name, volume(issue), pages, DOI. Must match database record.

---

## 2. Data Rules

### Rule 6: No batch generation
Max 5 papers per batch. Batch = batch hallucinations.

### Rule 7: Every number must be traceable
Effect sizes, sample sizes, percentages must come from the original paper, not LLM estimation.

### Rule 8: Evidence grade must match methodology
S = meta-analysis, A = RCT, B = quasi-experiment. No guessing.

---

## 3. Workflow Rules

### Rule 9: Run validation before adding papers
Always run the verification script before adding. No verification = no push.

### Rule 10: Roll back immediately on discovery
Delete files, git commit with reason, record in iron rules, notify owner.

### Rule 11: Honesty first
When called out, don't evade, don't blur, don't say 'possibly real'. Hallucination is hallucination.

## 4. Project Boundary Rules

### Rule 12: No programming
Documentation and analysis only. No executable scripts in this repo.

### Rule 13: No project merging
IR stays independent from Knowledge-Base and all other projects.

### Rule 14: Preserve directory structure
New files go into the correct 18-topic directory. No temp directories.

### Rule 15: Don't modify existing files
Add only, never edit or delete unless explicitly requested.

---

## 5. Technical Execution Rules

### Rule 16: Don't write .py files through PowerShell echo
Use Python open() or save script files to disk. (ref: KB DO_NOT_REPEAT #1)

### Rule 17: No triple-quoted strings in python -c in PowerShell
PowerShell breaks quote parsing. Use temp script files. (ref: KB DO_NOT_REPEAT #2)

### Rule 18: Clean temp files before git commit
No db_backup_*, error_*, integration_*, __pycache__, .pyc. (ref: KB DO_NOT_REPEAT #3)

### Rule 19: Explicit file encoding
Use UTF-8. Write with encoding=utf-8. Read with utf-8 then utf-8-sig fallback.

### Rule 20: Check file existence before operations
Check before write. Confirm before delete. Backup before overwrite.

### Rule 21: Check command results after each step
Verify return code and output before proceeding.

### Rule 22: Check git status before git operations
git status before every commit/push. (ref: KB LESSONS_LEARNED Git #1)

### Rule 23: Don't assume tool availability
Verify tool/command exists before use. Have fallback for failure.

### Rule 24: File paths with spaces in PowerShell
Quote paths properly. Use Python for path handling instead of PowerShell.

---

## 6. Behavior Rules

### Rule 25: Ask before act
For delete, move, overwrite, or batch operations: explain scope and get confirmation first.

### Rule 26: No fabricated output
Never fake data, citations, or analysis to look complete. Better empty than fake.

### Rule 27: One paper at a time, verify each
Every operation on the paper database must be traceable to its commit.

### Rule 28: Understand context before acting
Read the review doc, README, and 3 existing papers in the target directory before adding.

## 7. Known Environment Issues

| Issue | Solution |
|---|---|
| PowerShell quoting breaks Python -c multiline strings | Use temp .py files instead |
| Unstable PyPI/GitHub network (China) | SSH git, pip retry logic |
| Windows long path issues (node_modules etc) | Ensure .gitignore covers them |
| Unstable GitHub clone from China | ZIP via API more reliable than git clone |
