# 01 捕获系统设计

> 子系统：01-Capture-from-CC-FileTools  
> 对应 A 线模块：文档导入与资料采集  
> 参考实现：Claude Code FileReadTool / FileEditTool / FileWriteTool  
> 关联知识库：09-Learning-Science-Knowledge-Base (07_全球资源、08_工具系统)

## 一、系统定位

### 1.1 捕获系统在整个学习 OS 中的角色

捕获系统是 A-Line Human Learning OS 的第一个活性子系统，承担着**「学习原料的数字化入口」**这一核心角色。在人类学习的完整链路中，捕获位于最上游——任何知识要进入个人学习系统，首先要经过捕获这一关。

整个 A-Line 的学习流水线可以概括为：

```
捕获 → 诊断 → 计划 → 记忆 → 反馈 → 迭代
  ↑                                      ↓
  └────────── 知识库双向同步 ──────────────┘
```

- **捕获（01）**：从文件、网页、笔记、录音、图像等各种来源采集原始学习资料
- **诊断（02）**：分析捕获到的资料，识别学习者的薄弱点与知识结构
- **计划（03）**：根据诊断结果制定学习计划
- **记忆（04）**：将资料转化为 Anki/SuperMemo 等间隔重复系统的记忆卡片
- **反馈（05）**：追踪学习效果并形成闭环
- **09 知识库**：捕获系统与知识库形成双向数据流——捕获的资料自动关联到知识库对应模块，知识库的反向索引又能辅助诊断系统识别知识缺口

### 1.2 与上游输入源的关系

捕获系统面向的输入源涵盖三大类别：

| 输入类型 | 来源示例 | 格式 | 处理难度 |
|---------|---------|------|:--------:|
| 结构文本 | PDF、Markdown、HTML、Word | 有结构 | 低-中 |
| 半结构文本 | 网页剪报、论坛帖子、邮件 | 混合格式 | 中 |
| 非结构素材 | 录音转写、OCR 结果、手写笔记 | 无结构 | 高 |
| 多媒体 | 课程视频截图、图表、数据表 | 二进制 | 高 |

捕获系统需要以统一的管道处理这些不同形态的资料，并将它们转化为知识库可消费的结构化数据。

### 1.3 与 Claude Code File Tools 的渊源

本系统的设计灵感来自 Claude Code 的 FileReadTool / FileEditTool / FileWriteTool。这三个工具在代码仓库语境中解决了「如何安全高效地读写文件」的问题；迁移到学习系统语境后，我们需要解决的是「如何安全高效地采集和导入各种学习资料」。核心设计理念一脉相承：

- **FileReadTool 的分块读取** → 学习资料的大文件自动分块阅读
- **FileEditTool 的增量编辑** → 学习笔记的增量更新、只处理变更部分
- **FileWriteTool 的写入校验** → 捕获记录的完整性和一致性保证

## 二、分块读取策略

### 2.1 设计动机

学习资料天然具有**规模差异性**的挑战。一篇博客文章可能只有几百字，一本电子书可能有数十万字，一个论坛帖子可能只有几段话。不分块读取时，大文件会导致以下问题：

1. **内存溢出**：一次性读入数万行文本可能耗尽资源
2. **编码/解码效率低**：大文件一次解码的容错率低
3. **处理速度慢**：后续的语义分析、关键词提取等步骤均需等文件读完后才能开始
4. **不利于增量处理**：无法定位「哪一部分是新的」

### 2.2 动态分块算法

参考 FileReadTool 的分块思路，捕获系统采用**自适应分块策略**：

```
输入文件 → 文件大小探测 → 选择分块策略 → 逐块处理 → 合并结果
```

**分块策略矩阵**：

| 文件大小 | 分块大小 | 重叠量 (overlap) | 策略说明 |
|---------|:--------:|:---------------:|---------|
| < 100KB | 不分块 | — | 一次读取，直接处理 |
| 100KB - 5MB | 32KB | 1KB | 中等分块，保证语义段落完整 |
| 5MB - 100MB | 128KB | 4KB | 大分块，适合教材/论文 |
| > 100MB | 512KB | 16KB | 超大分块，适合语料库/音视频转写文本 |
| 压缩文件(.zip/.rar) | 逐文件解压后 | — | 先解压再按上述规则分块 |

### 2.3 语义感知分块

纯字节大小的分块可能切断语义完整的段落。改进版采用**语义感知分块**，借鉴 FileReadTool 的「换行感知 + 缩进感知」策略：

1. **优先以自然段落为分块边界**：遇到连续两个换行符时切断
2. **Markdown/HTML 标签感知**：不切断 &lt;h1&gt;&lt;/h1&gt; 等标记结构
3. **列表和代码块完整性**：列表项和代码块不被跨块分割
4. **回退机制**：如果连续 50KB 内没有语义边界，则强制在 64KB 处切断

### 2.4 块级处理流水线

每个分块独立通过以下流水线：

```
分块 → 编码检测 → 清洗（去BOM/修复乱码） → 元数据提取 → 语义标签 → 写入暂存区
```

所有分块处理完成后，启动合并阶段：

```
暂存区 → 去重检测 → 顺序合并 → 全文索引 → 写入知识库
```

## 三、编码检测机制

### 3.1 为什么需要编码检测

学习资料来源多样，编码格式千差万别。仅中文环境就可能遇到：

- UTF-8（最普遍，现代文件）
- UTF-16（Windows 系统生成的文件）
- GBK/GB2312（简体中文传统编码）
- Big5（繁体中文的传统编码）
- Shift-JIS（日文资料）
- Latin-1 / Windows-1252（欧洲语言资料）
- ASCII / raw bytes（纯文本）

错误识别编码会导致乱码，进而使整个捕获流程失效。

### 3.2 多级编码检测引擎

参考 FileReadTool 的编码检测策略，构建设计三层检测引擎：

**第一层：BOM 检测（O(1) 时间）**

```
文件头部前 4 字节 → 匹配 BOM 签名：
  EF BB BF       → UTF-8
  FF FE          → UTF-16 LE
  FE FF          → UTF-16 BE
  00 00 FE FF    → UTF-32 BE
  FF FE 00 00    → UTF-32 LE
```

BOM 检测是最快的途径，但仅对含 BOM 的文件有效（约覆盖 30% 的场景）。

**第二层：启发式检测（O(n) 时间）**

对不含 BOM 的文件，使用统计学方法：

1. **UTF-8 有效性检查**：验证字节序列是否合法的 UTF-8 编码
2. **NULL 字节密度检测**：UTF-16 文件中每两个字节有一个 NULL（ASCII 字符区间）
3. **高频字节分布分析**：GBK 中中文字符的高位字节集中在特定范围
4. **字符集冲突解决**：多个编码都能解释时，选择「最不可能出现乱码字符」的那个

**第三层：人机协作确认**

当引擎的置信度低于 85% 时：

1. 在捕获记录中标记「编码待确认」
2. 将前 1KB 内容的两种编码解读同时展示给用户
3. 用户确认后记录偏好，下次同类文件自动使用该编码

### 3.3 多语言资料处理

捕获系统需支持多语言学习场景（参考知识库 05_学科专项 > 02_外语学习）：

| 语言 | 常见编码 | 特殊处理 |
|------|---------|---------|
| 中文（简） | UTF-8, GBK, GB2312 | 繁简转换可选 |
| 中文（繁） | UTF-8, Big5 | 繁简转换可选 |
| 日文 | UTF-8, Shift-JIS, EUC-JP | 假名-汉字边界标注 |
| 韩文 | UTF-8, EUC-KR | 字母分写 |
| 欧洲语言 | UTF-8, Latin-1, Windows-1252 | 变音符号保留策略 |
| 阿拉伯文 | UTF-8, Windows-1256 | 从右到左排序处理 |

### 3.4 编码转换与容错

```
原始文件 → 检测编码 → 转换为 UTF-8 → 校验有效性 → 记录原始编码元数据
                          ↓ 失败
                     逐字节容错 → 跳过非法字节 → 生成编码问题报告
```

转换后的文件统一使用 UTF-8 存储，原始编码信息保留在捕获记录的 metadata.originalEncoding 字段中，以便追溯。

## 四、增量处理流程

### 4.1 设计原则

增量处理借鉴 FileEditTool 的「只处理变更部分」哲学，核心目标是**避免重复工作**。在学习场景中，学习者常常：

- 在同一个讲义文件上反复添加注释
- 在笔记中补充新的理解和联系
- 在阅读进度表中标记已读部分

每次都重新处理完整文件不仅浪费计算资源，还会破坏已有的知识关联。

### 4.2 变更检测机制

```
会话启动时 → 获取文件列表 → 计算文件哈希（SHA256） → 与上次捕获记录比较
                                 ↓ 哈希一致
                            跳过该文件（无变更）
                                 ↓ 哈希不一致
                            计算新旧版本 diff
                                 ↓ diff 为空（仅元数据变化？）
                            标记为"元数据变更"不重新处理
                                 ↓ diff 非空
                            进入增量处理流水线
```

**哈希策略**：使用 SHA256 对整个文件计算，但为提高效率，对大文件缓存每 1MB 的分块哈希，变更时仅重新计算变更块。

### 4.3 增量处理流水线

```
变更文件 → 分块级 diff → 仅处理变更分块 →
  ↓
提取变更内容中的: 新知识点 / 新连接 / 新问题 / 新注释
  ↓
更新知识库索引（不重建已有索引）
  ↓
触发相关模块的增量更新通知
  ↓
记录本次变更到捕获日志
```

### 4.4 冲突处理

当同一资料的多个版本同时产生变更时：

| 场景 | 策略 |
|------|------|
| 本地修改 vs 上次已捕获版本 | 以本地最新版本为准 |
| 同一资料被多设备修改 | 时间戳后来者优先，冲突标记待审 |
| 用户手动撤回变更 | 保留上一版本快照，恢复机制 |

## 五、与 09 知识库的对接

### 5.1 对接架构

捕获系统的核心产出需要无缝汇入 09 知识库的对应模块。整体架构如下：

```
捕获系统                       09 知识库
─────────                     ──────────
采集管道 → 自动分类器 → 匹配知识库模块 → 写入对应目录
  ↑                                ↓
用户手动指定                     更新全局索引
类别（可选）                         ↓
                             触发相关模块的关联更新
```

### 5.2 自动分类规则

捕获的资料通过以下规则自动判断应归入知识库的哪个模块：

| 捕获内容特征 | 目标知识库模块 | 匹配依据 |
|------------|:-------------:|---------|
| 学术论文/研究报告 | 01_通用学习科学 / 15_论文摘要库 | DOI 检测、标题关键词 |
| 学习方法论述 | 01_通用学习科学 → 具体方法 | 关键词匹配（如"间隔重复"→02_间隔重复） |
| 记忆技术案例 | 02_记忆技术 | 方法名称检测 |
| 学科知识点 | 05_学科专项 → 对应学科 | 学科关键词库 |
| 行业资料 | 06_行业专项 → 对应行业 | 行业词表 |
| 工具使用教程 | 08_工具系统 | 工具名称（如 Anki/Obsidian） |
| 论坛经验帖 | 07_全球资源 > 10_论坛经验采集库 | 来源域名匹配 |
| 教学方法 | 06_行业专项 > 01_教育培训 | 教学关键词 |
| 未分类内容 | 临时缓存区，等待用户确认 | — |

### 5.3 捕获记录与知识库的双向关系

每条捕获记录在知识库中生成一个对应的**资源条目**，包含：

```
捕获记录 ID ←→ 知识库条目 ID
    ↓               ↓
原始数据         结构化摘要
    ↓               ↓
元数据           关联学习方法
    ↓               ↓
变更历史         交叉引用
```

当知识库中的某个学习方法更新时（如发现间隔重复的新元分析），系统可以反向追踪到所有引用该方法的捕获资料，提示用户是否需要重新评估这些资料的价值。

### 5.4 对接流程的完整生命周期

```
1. 用户采集资料（文件/网页/笔记）
2. 捕获系统完成分块读取、编码检测
3. 自动分类器判断目标模块
4. 写入知识库对应目录
5. 更新知识库全局索引（15_交叉索引图）
6. 更新「14_90天执行路线图」中的资源清单
7. 返回答：新捕获了 X 篇资料，关联到 Y 个模块
8. 如果分类置信度 < 70%，等待用户人工确认
```

## 六、数据结构设计

### 6.1 核心捕获记录

每条捕获记录对应一个独立的资料项，使用 JSON 格式存储：

```json
{
  "captureId": "CAP-20260625-001",
  "captureType": "file | webpage | note | audio | image",
  "status": "pending | processed | error | user_review",
  
  "source": {
    "originalPath": "/path/to/原文件.pdf",
    "originalEncoding": "GBK",
    "fileSize": 5242880,
    "fileHash": "sha256:abc123...",
    "mimeType": "application/pdf",
    "sourceUrl": "https://...",
    "sourceDescription": "用户自定义来源说明"
  },
  
  "content": {
    "title": "资料标题（自动或手动）",
    "language": "zh-CN",
    "chunks": [
      {
        "chunkIndex": 0,
        "chunkSize": 32768,
        "chunkHash": "sha256:def456...",

    ],

    "ocrRequired": false
  },
  
  "version": {
    "captureVersion": 3,
    "changeHistory": [
      {"version": 1, "capturedAt": "2026-06-20T10:00:00Z", "changeType": "initial"},
      {"version": 2, "capturedAt": "2026-06-24T16:30:00Z", "changeType": "incremental", "summary": "新增注释 5 条"},
      {"version": 3, "capturedAt": "2026-06-25T12:00:00Z", "changeType": "incremental", "summary": "更新关键词"}
    ]
  },
  
  "quality": {
    "encodingConfidence": 0.98,
    "chunkIntegrity": "complete",
    "classificationConfidence": 0.92,
    "needsReview": false,
    "reviewNotes": ""
  }
}
```

### 6.2 字段设计说明

**captureId**：全局唯一标识，格式为 `CAP-{日期}-{当日序号}`，支持按时间范围查询。

**source 块**：完整记录文件来源信息，包括原始编码、哈希和类型。`fileHash` 用于增量处理的变更检测。

**content 块**：存储分块处理后的内容。每个 `chunk` 包含独立的哈希，方便增量时只替换变更块。`semanticBoundary` 记录分块边界类型，用于合并阶段的还原。

**classification 块**：记录自动分类的结果，`confidence` 字段为后续人工复审提供决策依据。`relatedMethods` 与知识库中的学习方法交叉引用。

**version 块**：追踪捕获记录的变更历史，支持回滚到任意版本。

### 6.3 索引设计

为支持高效查询，建立以下索引：

| 索引名称 | 键 | 用途 |
|---------|:--:|------|
| 主索引 | captureId | 单一记录查询 |
| 文件哈希索引 | source.fileHash | 变更检测 |
| 模块索引 | classification.targetModule | 按知识库模块聚合 |
| 时间索引 | content.processedAt | 时间范围查询 |
| 状态索引 | status | 待处理/出错 记录筛选 |
| 关键词全文索引 | classification.keywords | 语义搜索 |

## 七、与上游子系统的接口

### 7.1 捕获 → 诊断（子系统 02）

```
捕获完成事件 → 发送新资料摘要 → 诊断系统触发分析
     ↓                              ↓
captureId + content摘要        返回分析结果：薄弱点识别
     ↓                              ↓
等待诊断完成                   更新捕获记录的分析字段
```

### 7.2 捕获 → 记忆系统（子系统 04）

捕获系统为记忆系统提供卡片生成原料：

```
捕获资料 → 自动提取候选知识点 → 推送至 Palace(Memdir)
               ↓
标注优先级：高/中/低（基于分类置信度和资料类型）
```

### 7.3 捕获 → 反馈系统（子系统 05）

```
捕获记录 → 反馈系统追踪 → 学习效果与资料质量的关系分析
```

## 八、异常处理与容错

| 异常场景 | 处理策略 |
|---------|---------|
| 文件无法读取（权限/损坏） | 记录错误，跳过该文件，不中断管道 |
| 编码检测置信度低 | 使用 UTF-8 作为回退，标记待审 |
| 单个分块处理超时 | 跳过该块，继续下一个，最后统一重试 |
| 分类器无法匹配任何模块 | 存入临时区，标记「待分类」，每日汇总通知用户 |
| 磁盘空间不足 | 暂停捕获，清理临时缓存，发送告警 |
| 增量 diff 冲突 | 保留两个版本，标记冲突等待用户裁决 |

## 九、总结与下一步

捕获系统作为 A-Line 的入口，其质量直接决定了后续所有子系统的输入质量。从 Claude Code 的 FileReadTool 中借鉴的分块读取、编码检测、增量处理三大模式，为构建一个健壮、高效、支持多语言的学习资料采集管道提供了扎实的技术基础。

**下一步工作**：

1. 原型实现核心分块引擎和编码检测模块
2. 构建自动分类器（规则引擎 + NLP 辅助）
3. 实现捕获记录数据库和索引
4. 与 09 知识库的文件系统对接
5. 编写捕获管道的集成测试（处理各种编码和格式的测试套件）
