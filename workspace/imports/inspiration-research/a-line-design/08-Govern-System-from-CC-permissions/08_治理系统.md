## 一、系统定位

### 1.1 治理系统在学习 OS 中的角色

治理系统是 A-Line Human Learning OS 的**规则中枢和防线系统**，承担着「确保学习行为遵循科学原则、避免常见误区、动态匹配认知状态」的核心职责。在整个学习 OS 中，治理系统不像其他子系统那样直接处理具体的学习任务，而是**作为一个横切关注点（cross-cutting concern）嵌入所有学习活动**。

`
          捕获(01)   诊断(02)   计划(03)   训练(07)   记忆(04)
              +         +         +         +         +
              |         |         |         |         |
              +----+----+----+----+----+----+----+----+
                   |             治理(08)             |
                   |   (规则校验 / 模式切换 / 自检)    |
                   +----+----+----+----+----+----+----+
                        |         |         |         |
                   元认知(01-11)  避坑(12)  反馈(05)  画像(06)
`

治理系统在整个 OS 中的角色体现在三个维度：

**1. 规则执行者** —— 将学习科学原理转化为可执行的规则检查。例如「禁止仅划重点而不测试」「交错比例不得低于最低阈值」「连续训练超时强制休息」。

**2. 模式管理者** —— 根据学习者的目标（专注复习、常规巩固、严格考试）切换治理的严格程度，类似于 CC 系统的 PermissionMode。

**3. 自检监护人** —— 在元认知层面帮助学习者发现自己的认知偏差和学习误区，与 12_避坑区的知识深度联动。

### 1.2 与 CC 权限模型的渊源

本系统的设计灵感来自 Claude Code 的强制权限链。CC 的工具执行链是一个多层过滤的管道——输入验证、自检、Pre-Hooks、权限判断、执行、Post-Hooks，每一层都可能阻止或修改工具的行为。

| CC 组件 | 行数 | 核心职责 | A 线映射 |
|---------|:----:|---------|---------|
| Tool.ts | 754 | 工具基类和权限接口 | 学习规则定义基类 |
| PermissionContext.ts | 379 | 权限上下文管理 | 学习模式上下文 |
| permissionLogging.ts | 220 | 权限日志 | 学习行为审计日志 |
| toolExecution.ts | 1,745 | 工具执行链（含权限校验） | 学习规则引擎执行链 |

### 1.3 治理系统的核心设计原则

**原则一：无治理不学习** —— 任何学习行为在未经治理系统校验之前不得执行。这是从 CC「无权限不执行」继承的核心原则。

**原则二：治理等级可调** —— 不同的学习目标需要不同的治理严格度。「专注」模式给予最大自由，「考试」模式开启最严格检查。治理不应该是不可跳过的障碍，而应该是可感知的护栏。

**原则三：治理即反馈** —— 每一次规则违反都应产生有意义的反馈，而不仅仅是阻拦。「你不能这样做」的背后必须跟「因为你正处于 X 模式，而 Y 规则被触发，建议你改为 Z」。

**原则四：自检优先** —— 最有效的治理是学习者自己的治理。系统应逐渐训练学习者将外部规则内化为自我监控能力（元认知训练的核心目标）。

## 二、学习规则引擎

### 2.1 规则引擎架构

学习规则引擎的设计直接映射自 CC 的强制权限链。每个学习行为（练习、测试、阅读、复习等）都必须经过六层规则检查：

`
学习行为请求
      |
      v
+-------------+   Zod Schema 校验（输入验证）
|  ① Schema  |   → 行为的参数是否合法？格式是否正确？
|   Validator |   失败 -> 返回错误，提示正确格式
+------+------+
       |
       v
+-------------+   规则自检（准入条件）
|  ② Self-   |   → 当前规则集是否适用于此行为？
|   Check     |   跳过 -> 标记为「不适用」继续
+------+------+
       |
       v
+-------------+   Pre-Learning Hooks（预处理校验）
|  ③ Pre-    |   → 行为产生前的前置条件是否满足？
|   Learning  |   阻塞 -> 中断行为，说明阻塞原因
+------+------+
       |
       v
+-------------+   canExecute（核心规则判断）
|  ④ Rule    |   → 在当前位置/时间/状态下，此行为是否允许？
|   Judge     |   拒绝 -> 进入规则违反处理流程
+------+------+
       |
       v
+-------------+   实际学习行为执行
|  ⑤ Execute |
+------+------+
       |
       v
+-------------+   Post-Learning Hooks（后处理校验）
|  ⑥ Post-   |   → 行为执行后是否产生异常状态？
|   Learning  |   记录 -> 更新学习画像和日志
+------+------+
`

### 2.2 规则引擎核心实现

`python
class LearningRuleEngine:
    """学习规则引擎，管控所有学习行为"""
    
    def check_action(self, action: LearningAction, context: LearningContext) -> ActionResult:
        # Step 1: Schema 校验
        schema_result = self.validate_input(action)
        if not schema_result.is_valid:
            return ActionResult.REJECTED(
                reason=schema_result.error,
                phase="schema_validation",
                suggestion=schema_result.correction_hint
            )
        
        # Step 2: 规则自检
        if not self.self_check(action, context.mode):
            return ActionResult.SKIPPED(
                reason="当前规则集不适用此行为",
                phase="self_check"
            )
        
        # Step 3: Pre-Learning Hooks
        hook_result = self.execute_pre_hooks(action, context)
        if hook_result.is_blocking:
            return ActionResult.BLOCKED(
                reason=hook_result.blocking_reason,
                phase="pre_hooks",
                suggestion=hook_result.alternative
            )
        
        # Step 4: 核心规则判断
        rule_result = self.can_execute(action, context)
        if not rule_result.is_allowed:
            # 进入规则违反处理
            self.handle_violation(action, context, rule_result)
            return ActionResult.VIOLATED(
                reason=rule_result.reason,
                phase="rule_judgment",
                violated_rule=rule_result.rule_id,
                consequence=rule_result.consequence
            )
        
        # Step 5: 执行
        execution_result = self.execute_action(action, context)
        
        # Step 6: Post-Learning Hooks
        self.execute_post_hooks(action, context, execution_result)
        
        return ActionResult.ALLOWED(result=execution_result)
`

### 2.3 规则定义 DSL

规则引擎采用声明式规则定义语言，方便用户自定义学习规则：

`python
@learning_rule(
    rule_id="R001",
    name="禁止纯阅读",
    severity="warning | block",
    applies_to=["reading", "reviewing"],
    modes=["focus", "default", "exam"]
)
class NoPassiveReadingRule(LearningRule):
    """禁止不伴随主动回忆的纯阅读行为"""
    
    def condition(self, action, context):
        # 如果行为是「阅读」且没有伴随「测试」或「提取」
        return (action.type == "reading" 
                and not action.has_accompanying_test
                and action.duration_minutes > 10)
    
    def consequence(self, action, context):
        return RuleViolation(
            message="已连续阅读 10 分钟未进行任何主动提取",
            suggestion="建议立即尝试：合上材料，复述刚刚读到的三个要点",
            auto_remediation="强制弹出测试卡片"
        )
`

### 2.4 规则分类体系

学习规则按触发条件和覆盖范围分为五大类：

| 规则类别 | 触发条件 | 示例 | 默认严重度 |
|---------|---------|------|:---------:|
| 时间规则 | 时间维度超标或被违反 | 连续学习超过 90 分钟强制休息 | block |
| 质量规则 | 学习质量低于阈值 | 正确率低于 40% 时禁止进入下一章节 | warning |
| 方法规则 | 学习方法偏离科学原则 | 纯阅读超过 10 分钟未提取 | warning |
| 情境规则 | 环境或状态不匹配 | 认知疲劳指数 > 0.8 时禁止高难度任务 | block |
| 路径规则 | 学习顺序偏离计划 | 跳过了前置知识点直接学习依赖内容 | block |

### 2.5 规则优先级与冲突解决

`python
class RuleConflictResolver:
    """规则冲突解决器"""
    
    def resolve(self, rules: List[Rule], context: LearningContext) -> Rule:
        # 优先级 1：更高严重度的规则优先
        severity_order = {"block": 3, "warning": 2, "info": 1}
        
        # 优先级 2：更具体规则优先于通用规则
        specificity_order = sorted(
            rules,
            key=lambda r: (severity_order.get(r.severity, 0), 
                          r.specificity_score, 
                          r.mode_match_score(context.mode)),
            reverse=True
        )
        
        # 优先级 3：考试模式下 block 规则永远优先
        if context.mode == "exam":
            block_rules = [r for r in specificity_order if r.severity == "block"]
            if block_rules:
                return block_rules[0]
        
        return specificity_order[0]
`

## 三、三种学习模式

### 3.1 模式定义与映射

三种学习模式直接映射自 CC 的 PermissionMode。每一种模式代表了不同的治理严格度和自主权：

| 学习模式 | CC 映射 | 治理严格度 | 适用范围 | 核心特征 |
|---------|---------|:---------:|---------|---------|
| 专注模式 | bypassPermissions | 低 | 深度沉浸学习 | 最小干预，完全信任 |
| 复习模式 | default | 中 | 日常巩固复习 | 标准干预，交互式校验 |
| 考试模式 | plan | 高 | 严格自测和考验 | 最大干预，计划内允许 |

### 3.2 专注模式（Focus Mode）

专注模式是治理最宽松的模式，适用于深度沉浸式学习场景，如精读复杂材料、进行创造性思考、解决难题。对应 CC 的 ypassPermissions。

`python
class FocusMode(LearningMode):
    def __init__(self):
        self.severity_overrides = {
            "warning": "info",    # 警告降级为信息
            "block": "warning",   # 阻塞降级为警告
        }
        self.bypassed_rules = [
            "time_limit_soft",    # 软性时间限制 bypass
            "method_diversity",   # 方法多样性检查 bypass
            "topic_strictness",   # 主题严格度 bypass
        ]
    
    def should_block(self, violation: RuleViolation) -> bool:
        # 专注模式下只有极少数情况会真正阻塞
        return violation.severity == "block" and violation.is_hard_limit
`

**专注模式的治理特点**：

1. **最小化中断** —— 非必要不弹出提示。即使用户违反了某些方法规则（如长时间不提取），也仅记录日志，不打断学习流。
2. **信任用户判断** —— 在专注模式中，系统默认用户的判断优先于规则引擎的预设值。
3. **后台收集数据** —— 虽然是「bypass」模式，治理系统仍然在后台记录所有数据，用于模式切换后的分析和建议。
4. **智能触发条件** —— 专注模式通常自动触发。当诊断系统检测到用户处于「心流状态」或「高度专注」时，自动建议切换至此模式。

**专注模式适用场景**：
- 研读复杂论文或技术文档
- 进行创造性写作或编程
- 解决多步骤难题
- 首次学习高难度概念

### 3.3 复习模式（Default Mode）

复习模式是日常默认模式，适用于标准学习过程。对应 CC 的 default 模式。

`python
class DefaultMode(LearningMode):
    def __init__(self):
        self.rules_in_effect = ALL_RULES  # 全部规则生效
        self.require_confirmation = True  # 需要交互确认
        self.auto_adjust = True           # 允许自动调整
    
    def should_block(self, violation):
        # 按标准规则处理
        return violation.severity == "block"
    
    def handle_warning(self, violation, callback):
        # 标准处理：弹出交互式提示
        callback.prompt_user(
            title="学习建议",
            message=violation.message,
            suggestion=violation.suggestion,
            options=[
                ("采纳建议", "apply"),
                ("忽略本次", "ignore_once"),
                ("加入白名单", "whitelist"),
            ]
        )
`

**复习模式的治理特点**：

1. **交互式校验** —— 每次规则触发都向用户呈现，提供「采纳/忽略/白名单」三个选项。用户的选择会被系统记录，用于个性化调整。
2. **标准规则集** —— 所有默认规则全部生效。这是系统预配置的最完整规则集合。
3. **自动调整** —— 系统可以根据用户的历史选择频率自动调整规则的触发阈值。如果用户总是忽略某条规则，系统会逐步降低其触发频率。
4. **学习机会** —— 每次规则触发都是一次「元认知训练」的机会。系统会在提示中包含学习科学解释。

**复习模式适用场景**：
- 日常间隔重复复习
- 常规练习和测试
- 知识回顾和巩固
- 错题分析

### 3.4 考试模式（Exam Mode）

考试模式是治理最严格的模式，适用于自我测试、模拟考试和高要求评估。对应 CC 的 plan 模式。

`python
class ExamMode(LearningMode):
    def __init__(self):
        self.exam_rules = [
            "no_hints",             # 禁止提示
            "no_second_chance",     # 禁止重试
            "strict_timing",        # 严格计时
            "no_topic_switch",      # 禁止主题切换
            "no_break_override",    # 考试期间禁止中断
            "strict_sequence",      # 严格按顺序作答
        ]
        self.violation_consequences = {
            "hint_requested": "记录为 wrong_answer",
            "topic_switched": "考试终止",
            "time_exceeded": "强制提交",
        }
    
    def should_block(self, violation):
        return True  # 考试模式下所有违规都阻止
    
    def handle_violation(self, violation):
        if violation.type == ViolationType.TIME_EXCEEDED:
            self.force_submit(exam_session)
        elif violation.type == ViolationType.TOPIC_SWITCHED:
            self.terminate_exam(exam_session, reason="topic_deviation")
`

**考试模式的治理特点**：

1. **零容忍策略** —— 几乎所有规则违反都会导致对应后果。提示、重试、跳题等行为不被允许。
2. **模拟真实考试环境** —— 通过规则限制创造真实的考试压力，提高自测效果。
3. **事后分析** —— 考试结束后，治理系统会生成详细的「治理违规报告」，分析哪些规则被触发、触发动机是什么，帮助学习者了解自己的考试弱点。
4. **渐进准备** —— 从复习模式过渡到考试模式不是突变的，而是通过一个「预考试」阶段逐步激活更多规则。

**考试模式适用场景**：
- 模拟考试
- 知识点掌握度评估
- 限时测试
- 高要求自我检验

### 3.5 模式切换策略

`python
class ModeSwitchManager:
    """智能模式切换管理器"""
    
    def suggest_switch(self, context: LearningContext) -> Optional[ModeSuggestion]:
        suggestions = []
        
        # 检查是否需要切换到专注模式
        if (context.diagnosis.flow_state_probability > 0.7
                and context.current_mode != "focus"):
            suggestions.append(ModeSuggestion(
                target_mode="focus",
                reason="检测到心流状态，建议切换至专注模式减少中断",
                confidence=0.85
            ))
        
        # 检查是否需要切换到考试模式
        if (context.schedule.has_upcoming_exam
                and context.diagnosis.review_completeness > 0.8
                and context.current_mode == "default"):
            suggestions.append(ModeSuggestion(
                target_mode="exam",
                reason="考试即将到来且复习充分，建议切换至考试模式模拟自测",
                confidence=0.75
            ))
        
        # 检查是否需要从专注模式切回默认模式
        if (context.current_mode == "focus"
                and context.session.duration > 120  # 超过 2 小时
                and context.diagnosis.fatigue_index > 0.6):
            suggestions.append(ModeSuggestion(
                target_mode="default",
                reason="专注时间较长，建议切回默认模式进行休息检查",
                confidence=0.80
            ))
        
        return max(suggestions, key=lambda s: s.confidence) if suggestions else None
    
    def switch_mode(self, target_mode: str, context: LearningContext):
        if target_mode == "exam":
            # 进入考试模式前的准备流程
            self.prepare_exam_mode(context)
        elif target_mode == "focus":
            # 进入专注模式前的保真流程
            self.save_focus_state(context)
        
        context.current_mode = target_mode
        self.log_mode_switch(context, target_mode)
`

**模式切换的触发条件汇总**：

| 切换方向 | 自动触发条件 | 用户手动触发 |
|---------|------------|------------|
| default -> focus | 心流概率 > 0.7 或 专注度 > 85% | 用户主动请求 |
| focus -> default | 连续专注 > 2 小时 或 疲劳 > 0.6 | 用户主动请求 |
| default -> exam | 考试即将开始且复习完成度 > 80% | 用户主动请求 |
| exam -> default | 考试结束 | 自动切换 |
| focus -> exam | 不允许直接切换 | — |
| exam -> focus | 不允许直接切换 | — |

## 四、规则配置

### 4.1 配置架构

规则配置支持从用户简配到系统细配的多层结构：

`
用户自定义规则（最高优先级）
      |
用户配置文件 (user_rules.yaml)
      |
系统推荐规则（中等优先级）
      |
系统默认规则 (default_rules.yaml)
      |
硬编码核心规则（最低优先级，不可覆盖）
`

### 4.2 用户自定义规则设计

用户可以通过标准化的配置文件自定义学习规则：

`yaml

    name: "自定义休息间隔"

      modes: ["default", "exam"]

  - rule_id: "U002"
    name: "夜间学习限制"

    condition:
      time_range: ["22:00", "06:00"]  # 夜间时段
      activity: "high_difficulty"      # 高难度任务

      action: "block_with_suggestion"

    applies_to:
      modes: ["default", "exam"]

    name: "新概念交错检查"

    condition:

      consecutive_same_type: 5  # 连续同类型超过 5 个
    consequence:
      action: "warning"

    severity: "info"
`

### 4.3 规则模板库

系统提供预设规则模板，用户可以一键启用：

class RuleTemplateLibrary:
    def __init__(self):

`python
class LearningPitfallDetector:
    """学习误区检测器 — 对接 12_避坑区知识库"""
    
    def __init__(self):
        # 从 12_避坑区加载误区知识
        self.pitfalls = self.load_pitfalls_from_kb()
    
    def detect_fluency_illusion(self, session: StudySession) -> Optional[Warning]:
        """检测流利错觉：只划重点、反复阅读"""
        metrics = session.get_recent_metrics(minutes=20)
        
        # 条件：纯阅读时间 > 80% 且无任何测试行为
        if (metrics.passive_reading_ratio > 0.8
                and metrics.test_count == 0
                and metrics.total_duration > 10):
            return Warning(
                pitfall="fluency_illusion",
                message="您已连续阅读超过 10 分钟，未进行任何主动提取",
                evidence=f"纯阅读比例 {metrics.passive_reading_ratio:.0%}",
                kb_reference="12_避坑区/04_常见学习误区与识别指南#一",
                suggestion="尝试合上材料，复述刚才读到的三个核心要点"
            )
        return None
    
    def detect_collection_hoarding(self, profile: LearningProfile) -> Optional[Warning]:
        """检测收藏幻觉：收藏不等于学会"""
        collected = profile.get_collected_items(days=30)
        studied = profile.get_studied_items(days=30)
        
        if len(collected) > 20 and (len(studied) / len(collected)) < 0.3:
            return Warning(
                pitfall="collection_hoarding",
                message="您的收藏/学习比偏低，存在收藏幻觉风险",
                evidence=f"收藏 {len(collected)} 项，仅学习 {len(studied)} 项",
                kb_reference="12_避坑区/04_常见学习误区与识别指南#二",
                suggestion="建议今天从收藏清单中挑选 3 项完成学习"
            )
        return None
    
    def detect_watch_only(self, session: StudySession) -> Optional[Warning]:
        """检测只看不练：50/50 规则"""
        input_ratio = session.get_input_output_ratio()
        
        if input_ratio > 0.7:  # 输入占比超过 70%
            return Warning(
                pitfall="watch_only",
                message=f"输入/输出比为 {input_ratio:.0%}/{1-input_ratio:.0%}，输入过多",
                evidence=f"输入 {session.input_minutes}min / 输出 {session.output_minutes}min",
                kb_reference="12_避坑区/04_常见学习误区与识别指南#三",
                suggestion="遵循 50/50 规则：花同等时间实践和应用"
            )
        return None
    
    def detect_single_method(self, profile: LearningProfile) -> Optional[Warning]:
        """检测方法单一化：迷信单一学习方法"""
        method_diversity = profile.get_method_diversity_score()
        
        if method_diversity < 0.3:
            return Warning(
                pitfall="single_method",
                message="您近期的学习方法比较单一",
                evidence=f"方法多样性得分 {method_diversity:.2f}",
                kb_reference="12_避坑区/04_常见学习误区与识别指南#四",
                suggestion="不同内容需要不同方法：记忆用间隔重复，理解用费曼技巧"
            )
        return None
`

### 6.2 学习前自检

`python
class PreLearningSelfCheck:
    """学习前自检机制"""
    
    def perform_check(self, plan: StudyPlan) -> CheckResult:
        issues = []
        
        # 检查 1：目标清晰度检测
        if not self.is_goal_specific(plan):
            issues.append(CheckIssue(
                category="goal_clarity",
                severity="warning",
                message="学习目标不够具体",
                suggestion="建议使用 SMART 原则重新定义目标",
                kb_reference="09/01_通用学习科学/08_刻意练习"
            ))
        
        # 检查 2：前置知识检测
        missing_prereqs = self.check_prerequisites(plan.subject)
        if missing_prereqs:
            issues.append(CheckIssue(
                category="prerequisites",
                severity="block" if len(missing_prereqs) > 2 else "warning",
                message=f"缺失前置知识：{', '.join(missing_prereqs)}",
                suggestion="建议先完成前置知识的复习"
            ))
        
        # 检查 3：时间适配检测
        if not self.is_time_sufficient(plan):
            issues.append(CheckIssue(
                category="time_allocation",
                severity="warning",
                message=f"计划时间 {plan.allocated_minutes}min 可能不足",
                suggestion=f"预估需要至少 {plan.estimated_minutes}min"
            ))
        
        # 检查 4：方法匹配检测
        method_match = self.check_method_suitability(plan)
        if method_match.score < 0.5:
            issues.append(CheckIssue(
                category="method_mismatch",
                severity="info",
                message=f"所选方法 '{plan.method}' 与目标类型不匹配",
                suggestion=method_match.alternative
            ))
        
        return CheckResult(
            passed=len([i for i in issues if i.severity == "block"]) == 0,
            issues=issues,
            summary=f"自检完成：发现 {len(issues)} 个问题（{sum(i.severity == 'block' for i in issues)} 个阻塞）"
        )
`

### 6.3 学习中自检

学习中自检是持续进行的，通常不需要用户主动触发：

`python
class InLearningSelfCheck:
    """学习中持续自检机制"""
    
    def monitor_and_check(self, session: StudySession):
        while session.is_active:
            # 检查 1：专注度检测
            focus = self.measure_focus(session)
            if focus < 0.4:
                self.report_issue(CheckIssue(
                    category="focus_lost",
                    severity="warning",
                    message="专注度下降至 40% 以下",
                    suggestion="建议暂停 2 分钟，做深呼吸后继续"
                ))
            
            # 检查 2：理解度检测
            comprehension = self.measure_comprehension(session)
            if comprehension < 0.5 and session.items_attempted > 5:
                self.report_issue(CheckIssue(
                    category="comprehension_gap",
                    severity="warning",
                    message=f"当前理解度 {comprehension:.0%}，连续出错",
                    suggestion="建议降级难度或切换学习方法"
                ))
            
            # 检查 3：疲劳度检测
            fatigue = self.measure_fatigue(session)
            if fatigue > 0.8:
                self.report_issue(CheckIssue(
                    category="fatigue_high",
                    severity="block" if fatigue > 0.9 else "warning",
                    message=f"疲劳指数 {fatigue:.0%}，已达高疲劳状态",
                    suggestion="建议立即休息 15 分钟"
                ))
            
            # 检查 4：方法有效性检测
            effectiveness = self.measure_method_effectiveness(session)
            if effectiveness < 0.3 and session.elapsed_minutes > 15:
                self.report_issue(CheckIssue(
                    category="method_ineffective",
                    severity="info",
                    message=f"当前方法效果不佳（{effectiveness:.0%}）",
                    suggestion=self.suggest_alternative_method(session)
                ))
            
            time.sleep(120)  # 每 2 分钟检查一次

class PostLearningSelfCheck:

class MetaCognitionTrainer:
    """基于自检结果的元认知训练"""
    
    def generate_reflection_prompt(self, check_result: CheckResult, mode: str):
        """根据自检结果生成反思提示"""
        if mode == "exam":
            return None  # 考试模式不干扰反思
        
        issues = check_result.issues

                type="positive_reinforcement",
                message="本次无自检问题！请思考：你的哪些学习习惯在今天特别有效？"
            )

            type="guided_reflection",
            message=f"本次自检发现的最主要问题是：「{top_issue.message}」",
            guiding_questions=[
                "你认为这个问题的根本原因是什么？",

                "这个问题与之前的知识库中的哪个误区有关？",
            ],

        )
    
    def track_metacognitive_growth(self, user_id, interval_days=7):

        history = self.get_check_history(user_id, days=interval_days)
        
        return GrowthReport(
            self_check_frequency=history.total_checks,
            issue_self_detection_rate=history.self_detected / max(history.total_issues, 1),
            improvement_rate=history.issues_resolved / max(history.issues_identified, 1),
            trend="improving" if history.improvement_rate > 0.6 else "needs_attention"

## 七、总结与下一步

治理系统是 A-Line 中身份最独特的子系统——它不直接参与学习，却渗透到每一个学习行为中。从 CC 权限模型的六层强制校验出发，治理系统构建了一套覆盖规则引擎、模式管理、规则配置、误区防护和自我检查的完整防线体系。

**系统设计要点回顾**：

1. **六层规则校验架构**（Schema → 自检 → Pre-Hooks → 核心判断 → 执行 → Post-Hooks）确保每个学习行为都经过科学原则的检验
2. **三种学习模式**（专注/复习/考试）对应不同的治理严格度，实现从最小干预到最大干预的灵活切换
3. **用户规则配置**通过标准的规则 DSL 和模板库，让用户可以根据个人需求定制学习规则
4. **与 12_避坑区的深度对接**将流利错觉、收藏幻觉、只看不练、方法单一化等常见误区转化为可执行的检测规则
5. **三层自检机制**（学习前/中/后）将元认知理论落地为持续的自查流程，培养学习者的自我监控能力
